# Web Deployment

> **WIP.** Server-side rendering and the HTTP server work today.
> Client-side WASM transpilation is stubbed (Phase 0): every callback
> currently round-trips to the server. The asset URLs, route mapping,
> and JS loader are stable and won't change shape when WASM lands.

The same binary that opens a desktop window can run as an HTTP server.
Set the environment variable and Azul replaces the platform shell with a
TCP listener:

```sh
AZ_BACKEND=web://127.0.0.1:8080 ./my-azul-app
```

The layout callback runs natively in the server process. Its output is
rendered to HTML with a `<style>` block built from the resolved CSS
cascade, then served at `/`. Images and fonts are extracted and served
under `/az/`.

## `AZ_BACKEND` URL format

Parsed by `parse_web_url` in `dll/src/web/config.rs:17`. Accepted forms:

| URL | Bind address |
|---|---|
| `web://127.0.0.1:8080` | localhost only |
| `web://0.0.0.0:3000` | all IPv4 interfaces |
| `web://[::1]:8080` | IPv6 loopback |
| `web://0.0.0.0:443?tls=cert.pem` | query string ignored today |

The `web://` prefix is case-insensitive. Anything after `?` is reserved
for future flags and silently dropped. A malformed value falls back to
the desktop shell.

## What runs server-side, what runs client-side

| Stage | Where it runs | Status |
|---|---|---|
| Layout callback (`Dom` construction) | server | works |
| CSS cascade → `StyledDom` | server | works |
| HTML serialization | server | works |
| Image / font collection | server | works |
| Browser layout, paint | client | works (browser does it) |
| `:hover` / `:focus` styling | client | works (CSS only) |
| Event callbacks | server (POST round-trip) | Phase 0 — see below |
| Framework WASM (`azul-mini.wasm`) | client | stubbed — 8-byte module |
| User-callback WASM | client | stubbed — none discovered |

Azul's full cascade resolves *all* conditional rules on the server
(theme, viewport, container, language). Only interactive pseudo-states
remain as CSS rules in the served stylesheet — those work without a
round-trip because the browser handles them.

## Pre-render, then serve

Startup is a five-phase pipeline in `dll/src/web/mod.rs:45`. Phases A–C
are stubbed; D and E are functional:

1. **A — Classify** (`classify.rs`). Walks the embedded `api.json` and
   tags each Azul C function as `Framework`, `ServerEntryPoint`, or
   `ReplaceWithDomPatcher`. The classification feeds the WASM lift
   pipeline.
2. **B — Mini WASM** (`mini_gen.rs`). Lifts framework functions into
   `azul-mini.wasm`. Phase 0 returns an 8-byte stub: `\0asm` magic +
   version 1.
3. **C — Callback discovery** (`cb_gen.rs`). Walks the DOM for
   registered callbacks and transpiles each. Phase 0 returns
   `Vec::new()`.
4. **D — Render routes**. For every registered route, calls its layout
   callback, runs the cascade, walks the `StyledDom`, and emits HTML +
   CSS rules. Image/font IDs are rebased so they don't collide across
   routes.
5. **E — HTTP server** (`server.rs`). One `std::net::TcpListener` thread
   accepts connections; each handler runs on a `std::thread::spawn`.

Routes registered on `AppConfig.routes` produce one pre-rendered page
each. Without any routes, the root window's layout callback becomes
`/`.

## Route patterns

Routes use `:param` segments matched by `azul_core::resources::match_route`:

```rust,ignore
# use azul_core::resources::Route;
# use azul_core::callbacks::LayoutCallback;
# fn user_page() -> LayoutCallback { unimplemented!() }
# fn product_page() -> LayoutCallback { unimplemented!() }
let routes = vec![
    Route { pattern: "/users/:id".into(), layout_callback: user_page() },
    Route { pattern: "/products/:sku".into(), layout_callback: product_page() },
];
```

The server tries exact match first, then parameterized match, then
falls back to `/`. Phase 0 serves the same pre-rendered template for
every parameter value — re-rendering with bound `RouteMatch` params is
not wired through the request handler yet (`server.rs:212`).

## Asset URL layout

Every URL under `/az/` is generated by `html_render.rs` and served by
the request router:

| URL | Content | Cache header |
|---|---|---|
| `/` and route patterns | pre-rendered HTML | none |
| `/az/loader.js` | bootstrap script | none |
| `/az/mini.{hash}.wasm` | framework WASM | `immutable, max-age=1y` |
| `/az/cb/{name}.{hash}.wasm` | per-callback WASM | `immutable, max-age=1y` |
| `/az/img/{id}` | PNG-encoded image | `immutable, max-age=1y` |
| `/az/font/{id}` | TTF font bytes | `immutable, max-age=1y` |
| `POST /az/exec/{node_id}` | run callback, return new HTML | none |

`{hash}` is an FNV-1a 64-bit content hash so a build refresh
invalidates the browser's cache automatically. The numeric `{id}` for
images and fonts is assigned sequentially during render. POST bodies
are capped at 16 MB; oversized payloads return `413 Payload Too Large`
(`dll/src/web/server.rs:132`).

## CSS emission

Azul runs the cascade server-side and emits one rule per node:

```css
#az_0 { display: flex; flex-direction: column; padding: 12px; }
#az_1 { color: #333; font-size: 14px; }
#az_1:hover { color: #000; }
```

Base styles come from `cache.computed_values` (every condition already
resolved). Pseudo-state overrides (`:hover`, `:focus`, `:active`,
`:checked`, `:disabled`) come from `cache.css_props` and are emitted as
separate selectors. The browser does the rest. Mapping lives in
`dll/src/web/html_render.rs:369`.

Bundled fonts in `AppConfig.bundled_fonts` are emitted as `@font-face`
rules pointing at `/az/font/{id}`. System font fallback is not exposed
to the browser — the server only emits fonts it has the bytes for.

## The loader script

`/az/loader.js` (generated by `loader_js.rs`) attaches three behaviors
to the served HTML:

1. **Callback wiring**. For each element with `data-az-cb` and
   `data-az-ev`, install an event listener that POSTs to
   `/az/exec/{cbId}` and replaces the document with the response.
2. **SPA-style navigation**. Internal `<a href="/...">` clicks are
   intercepted and routed through `fetch()` → `history.pushState` →
   document replace. Asset URLs (`/az/...`) are excluded.
3. **Back/forward**. A `popstate` listener re-fetches `location.pathname`.

The current implementation uses `document.open(); document.write(html);
document.close()` to swap the page. This destroys the running script,
which is fine for full-page replacement; future incremental updates
will replace it with `documentElement.innerHTML` or DOM diffing.

## Server-side callback execution

Today, every event runs on the server:

```
[browser]  click → POST /az/exec/3 + JSON event payload
[server]   look up callback for node 3 (TODO — not implemented)
[server]   run layout, render HTML
[server]   200 text/html
[browser]  document.write(response)
```

The `server.rs:191` handler reads the POST body, calls
`re_render_body()` which re-runs the *root* layout callback, and
returns the resulting HTML. The `node_id` is parsed but not yet
dispatched — this means every POST currently re-renders without
executing the targeted callback. A real dispatch pass will land before
Phase 0 closes.

Plan for Phases 1–3:

1. **Phase 1**: dispatch `node_id` to the registered callback on the
   server, pass through the JSON event payload, and apply the
   `Update::RefreshDom` result before re-render.
2. **Phase 2**: ship a real `azul-mini.wasm` so DOM construction,
   cascade, and hit-testing happen in the browser. POSTs disappear for
   pure-style updates.
3. **Phase 3**: lift user callbacks via the `Transpiler` trait (`transpiler.rs:39`).
   Callbacks that lift cleanly run client-side; the rest fall back to POST.

## Transpiler limits

The `Transpiler` trait (`transpiler.rs:39`) is the seam where the WASM
backend will plug in. The default is `StubTranspiler` — every
`lift_function` call returns `TranspileError`. This is by design: the
server-side fallback is a complete (if slow) implementation, so the
backend ships before remill-rs integration.

When the real transpiler lands, expect these constraints:

- **Pure callbacks lift; FFI callbacks don't.** A callback that uses
  only Azul C-API functions and `RefAny` operations can be lifted.
  Calls into platform APIs (file pickers, clipboard, threads, OS
  dialogs) cannot — they'll stay server-side.
- **Non-deterministic Rust features fall through.** `std::env`,
  `std::fs`, `std::process`, `Instant::now()` and similar have no WASM
  equivalent inside the sandbox; their callers fall back to POST.
- **Blocking code falls through.** WASM in browsers is single-threaded
  on the main loop; long-running callbacks must use `Task` / `Thread`
  on the server side, which means POST.
- **Function size matters.** The lifter takes `(name, addr, size)`
  triples (`transpiler.rs:46`). A callback that pulls in a 5 MB
  transitive symbol set will compile slowly and bloat the served
  bundle.

`is_available()` on the transpiler reports whether lifting is
functional. Until it returns `true`, all interaction round-trips.

## Deploying

The server has no built-in TLS, no auth, and no load balancing. It is a
standalone HTTP listener with `Connection: close` semantics. For a
public deployment, front it with a reverse proxy that handles those
concerns (nginx, Caddy, traefik):

```
[client] ──HTTPS──> [Caddy/nginx] ──HTTP──> [azul app on 127.0.0.1:8080]
```

A minimal Caddyfile:

```
my-app.example.com {
    reverse_proxy 127.0.0.1:8080
    encode gzip
}
```

Bind to `127.0.0.1` (not `0.0.0.0`) when behind a proxy on the same
host. Bind to `0.0.0.0` only inside a container or behind an
infrastructure firewall. Port 0 is not supported — the parser requires
an explicit port.

Resource considerations:

- **Memory**: the server keeps the entire pre-rendered HTML, all
  collected images, and all collected fonts in RAM. A multi-route app
  with large images can grow quickly; rebase + dedupe across routes is
  not implemented (`mod.rs:140`).
- **Concurrency**: one OS thread per connection. Fine for tens of
  concurrent users; for more, run multiple processes behind the proxy
  and share state externally (Phase 0 has no built-in shared state).
- **State**: `app_data` lives behind a single `Arc<Mutex<RefAny>>`
  (`server.rs:43`). Every POST locks it; callback-heavy workloads
  serialize through that mutex.

Browser support: the served HTML is plain HTML5 with no WASM
requirement in Phase 0. Anything that runs ES5 + `fetch()` works,
which covers all evergreen browsers.
