# Scrolling and Drag-and-Drop

> WIP. Scrolling is functional. Drag-and-drop has its data and effect types in place but the `DragEnter` / `DragOver` / `DragLeave` / `Drop` events are not yet generated by the event loop — see [Drag-and-drop status](#drag-and-drop-status). The page covers both subjects because they share the input pipeline.

Scrolling and drag are the two ways a pointer interacts with the *layout* rather than with a single element. Both are gestures: the framework tracks the mouse across multiple events, decides which gesture is in progress, and emits high-level events (`Scroll`, `DragStart`, …) that you handle the same way as any other event filter from [events](events.md).

## Making a node scrollable

CSS does the work. Set `overflow` to `scroll`, `auto`, or `hidden` on a node whose content exceeds its box, and the framework gives it scrollbars and wires the wheel/trackpad/touchpad events to it.

```rust,no_run
# use azul::prelude::*;
let scroll_box = Dom::create_div()
    .with_inline_style(
        "width: 400px; height: 200px; overflow-y: scroll;".into()
    );
```

The container's *scroll clip size* is its inner box (border-box minus borders, padding, and scrollbar track). The *content size* is the total extent of its children. Scrolling shifts which slice of the content sits inside the clip; the scrollbar reports `clip / content` as the thumb-size ratio. Both numbers are computed by the layout solver and stored on the node.

| Value of `overflow` | Effect |
|---|---|
| `visible` (default) | Children draw outside the box. No clip, no scrollbar, no `Scroll` events. |
| `hidden` | Children clip to the box. No scrollbar. No scrolling. |
| `scroll` | Children clip. Scrollbar always visible, even if not needed. Wheel scrolls. |
| `auto` | Children clip. Scrollbar visible only when the content overflows. Wheel scrolls. |

`overflow-x` and `overflow-y` set the axes independently. The shorthand `overflow: scroll` sets both.

### What you do not get

- **No viewport-level scrolling.** The window itself does not scroll. The `<html>` element is sized to the window (`height: 100%` via UA CSS). Make `<body>` or a descendant the scroll container — see `core/src/ua_css.rs:802` for the rationale.
- **No automatic IME-aware scroll-into-view** on focus. You scroll with `info.scroll_to(...)` from a focus callback if you want this.

## Reading scroll events

`HoverEventFilter::Scroll` fires for wheel/trackpad/touch scroll over a scrollable node. `ScrollStart` and `ScrollEnd` bracket a contiguous gesture so you can debounce work.

```rust,no_run
# use azul::prelude::*;
# extern "C" fn on_scroll(_: RefAny, _: CallbackInfo) -> Update { Update::DoNothing }
# fn build(data: RefAny) -> Dom {
let mut node = Dom::create_div();
node.add_callback(
    EventFilter::Hover(HoverEventFilter::Scroll),
    data,
    on_scroll,
);
# node
# }
```

Inside the callback, query the current scroll state through `CallbackInfo`. The two read APIs:

```rust,ignore
impl CallbackInfo {
    pub fn get_scroll_offset(&self) -> Option<LogicalPosition>;
    pub fn get_scroll_offset_for_node(&self, dom_id: DomId, node_id: NodeId)
        -> Option<LogicalPosition>;
    pub fn get_scroll_node_info(&self, dom_id: DomId, node_id: NodeId)
        -> Option<ScrollNodeInfo>;
}
```

`get_scroll_offset()` returns the offset for the hit node — convenient inside a `Scroll` callback. `get_scroll_node_info()` returns the full snapshot:

```rust,ignore
pub struct ScrollNodeInfo {
    pub current_offset: LogicalPosition,
    pub container_rect: LogicalRect,        // the visible viewport
    pub content_rect: LogicalRect,          // total scrollable extent
    pub max_scroll_x: f32,
    pub max_scroll_y: f32,
    pub overscroll_behavior_x: OverscrollBehavior,
    pub overscroll_behavior_y: OverscrollBehavior,
    pub overflow_scrolling: OverflowScrolling,
}
```

Defined at `layout/src/managers/scroll_state.rs:371`. `current_offset` is the live offset (may still be animating); `max_scroll_x` and `max_scroll_y` are the clamps.

## Scrolling programmatically

Two methods on `CallbackInfo`. Both queue a `CallbackChange::ScrollTo` that the runtime applies after the callback returns.

```rust,ignore
impl CallbackInfo {
    pub fn scroll_to(&mut self,
        dom_id: DomId,
        node_id: NodeHierarchyItemId,
        position: LogicalPosition);

    /// For rubber-banding / overscroll. Bypasses the [0, max] clamp.
    pub fn scroll_to_unclamped(&mut self,
        dom_id: DomId,
        node_id: NodeHierarchyItemId,
        position: LogicalPosition);
}
```

`position` is in the scroll container's content space — `(0, 0)` scrolls to the top-left of the content. To scroll a child into view, use `scroll_into_view(node_id, options)` on `CallbackInfo` (W3C `scrollIntoView` semantics; see `layout/src/callbacks.rs` for the full options struct).

To find the scrollable ancestor of an arbitrary node — for instance, the container you should scroll when a user interacts with a deep child:

```rust,ignore
pub fn find_scroll_parent(&self, dom_id: DomId, node_id: NodeId) -> Option<NodeId>;
```

Walks up from `node_id` to the nearest node registered with the scroll manager.

## How scroll events reach the right node

The pipeline:

1. The platform shell delivers a wheel/scroll event with a hit position.
2. Hit-testing (`layout/src/hit_test.rs`) resolves which DOM nodes are under the cursor, in front-to-back order.
3. The event manager finds the deepest scrollable ancestor of the hit node and routes the delta to it.
4. The scroll manager updates `current_offset` (clamped to `max_scroll_*` unless overscroll is in flight) and emits `Scroll`, `ScrollStart`, or `ScrollEnd` against the scrolled node.
5. Your callback fires with the standard `CallbackInfo`.

The cursor-hit logic that decides which node the gesture targets is at `layout/src/hit_test.rs:55` (`CursorTypeHitTest::new`). It walks `hit_test.hovered_nodes` and picks the *frontmost* node with an explicit CSS `cursor` (or a text child whose cursor it inherits).

## Smooth and momentum scrolling

The `ScrollManager` (`layout/src/managers/scroll_state.rs`) animates between scroll positions when `scroll_to` is called. Trackpad and wheel deltas are accumulated frame-to-frame; the reserved `SCROLL_MOMENTUM_TIMER_ID` (see [timers](timers.md#timerid--the-handle)) drives the inertia phase after a fling.

You don't manage the timer. CSS `-azul-overflow-scrolling: touch` enables momentum on a node; `overscroll-behavior: contain` prevents scroll chaining to the parent.

## Hit-test order

Front-to-back. `hit_depth = 0` is the topmost element under the cursor; deeper nodes have higher depths. A scroll routes to the topmost scrollable ancestor of the hit node, not to whichever scroll container the cursor happens to sit inside. This matches browser behaviour: a button's `cursor: pointer` overrides the container's `cursor: text`, and a `<select>` swallows wheel events that would otherwise scroll the page.

## Drag and drop — the model

The drag system is a discriminated union over the things a drag can be doing — selection, scrollbar thumb, DOM node, window move, window resize, OS file drop. All live in `core/src/drag.rs`:

```rust,ignore
pub enum ActiveDragType {
    TextSelection(TextSelectionDrag),
    ScrollbarThumb(ScrollbarThumbDrag),
    Node(NodeDrag),
    WindowMove(WindowMoveDrag),
    WindowResize(WindowResizeDrag),
    FileDrop(FileDropDrag),
}
```

The framework handles `TextSelection`, `ScrollbarThumb`, `WindowMove`, and `WindowResize` for you. `Node` and `FileDrop` are the variants you write callbacks for.

### Marking a node draggable

`draggable="true"` on the DOM node. The attribute parses; the gesture manager sees it during hit-test and starts a `NodeDrag` when the user begins a drag.

```rust,no_run
# use azul::prelude::*;
let card = Dom::create_div()
    .with_attribute(AttributeType::Draggable(true))
    .with_inline_style("padding: 12px; background: #fef;".into());
```

### Drag events

Defined in `core/src/events.rs`:

| Filter | Fires on | Purpose |
|---|---|---|
| `HoverEventFilter::DragStart` | source node | Drag begins. Set drag data here. |
| `HoverEventFilter::Drag` | source node | Each cursor move during drag. |
| `HoverEventFilter::DragEnd` | source node | Drag ends (drop or cancel). Read final `dropEffect`. |
| `HoverEventFilter::DragEnter` | target node | Cursor enters a candidate drop zone. |
| `HoverEventFilter::DragOver` | target node | Cursor stays over a drop zone (throttled). |
| `HoverEventFilter::DragLeave` | target node | Cursor leaves a drop zone. |
| `HoverEventFilter::Drop` | target node | User releases on a drop zone that has accepted the drop. |
| `HoverEventFilter::DroppedFile` | hit node | OS file-drop landed. |

Each variant has a corresponding `FocusEventFilter::*` and `WindowEventFilter::*` (see `core/src/events.rs:1158`). Use `Hover` for "this specific node is the drop zone"; use `Window` to observe drags anywhere in the window.

### Drag data — set on start, read on drop

`DragData` is a MIME-keyed payload. Mirrors the W3C `DataTransfer` API.

```rust,ignore
impl CallbackInfo {
    /// W3C dataTransfer.setData(type, data). Call from a DragStart callback.
    pub fn set_drag_data(&mut self, mime_type: AzString, data: Vec<u8>);

    /// W3C dataTransfer.types — visible during DragOver.
    pub fn get_drag_types(&self) -> Vec<AzString>;

    /// W3C dataTransfer.getData(type) — only readable inside the Drop handler.
    pub fn get_drag_data(&self, mime_type: &str) -> Option<Vec<u8>>;
}
```

Multiple MIME types per drag are allowed — set the same payload as `text/plain` for foreign drop targets and as your own `application/x-myapp-task` for the structured drop:

```rust,ignore
extern "C" fn on_drag_start(_: RefAny, mut info: CallbackInfo) -> Update {
    info.set_drag_data("text/plain".into(), b"Task #42".to_vec());
    info.set_drag_data("application/x-myapp-task".into(), task_id.to_le_bytes().to_vec());
    Update::DoNothing
}
```

### Accepting (or rejecting) a drop

Drop targets opt in. Inside a `DragEnter` or `DragOver` callback, inspect `get_drag_types()`; if the data is for you, call `accept_drop()` and set the drop effect:

```rust,ignore
impl CallbackInfo {
    /// Equivalent to event.preventDefault() in a W3C dragover handler.
    pub fn accept_drop(&mut self);

    /// W3C dataTransfer.dropEffect = "move" | "copy" | "link" | "none".
    pub fn set_drop_effect(&mut self, effect: DropEffect);
}
```

A node that does not call `accept_drop()` is not a drop target — the cursor shows the no-drop indicator and `Drop` will not fire on it. This matches the W3C model exactly.

```rust,ignore
extern "C" fn on_drag_over(_: RefAny, info: CallbackInfo) -> Update {
    if info.get_drag_types().iter().any(|t| t.as_str() == "application/x-myapp-task") {
        let mut info = info;
        info.accept_drop();
        info.set_drop_effect(DropEffect::Move);
    }
    Update::DoNothing
}
```

### Drop and DragEnd

`Drop` fires once on the accepted target with the drop position and the data. `DragEnd` fires once on the source — even if the drop was cancelled or rejected — with the final `dropEffect` so the source can finish the move/copy/link operation:

```rust,ignore
extern "C" fn on_drop(data: RefAny, info: CallbackInfo) -> Update {
    let payload = info.get_drag_data("application/x-myapp-task");
    if let Some(bytes) = payload {
        // ... move the task into the drop column ...
    }
    Update::RefreshDom
}

extern "C" fn on_drag_end(data: RefAny, info: CallbackInfo) -> Update {
    // Inspect info.get_drop_effect() to know whether this was Move, Copy, Link, or None.
    Update::RefreshDom
}
```

### DropEffect and DragEffect

Two related enums in `core/src/drag.rs`:

| Type | Meaning | Variants |
|---|---|---|
| `DragEffect` | What the source allows. Set on the drag context at `DragStart`. | `Uninitialized`, `None`, `Copy`, `CopyLink`, `CopyMove`, `Link`, `LinkMove`, `Move`, `All` |
| `DropEffect` | What the target chose. Set in `DragOver` / `DragEnter`. | `None`, `Copy`, `Link`, `Move` |

A drop target's `DropEffect` must be in the source's `DragEffect` set, otherwise the drop is rejected.

## Drag-and-drop status

Per the design doc `scripts/DRAG_DROP_REPORT.md`, three pieces of the drag system were planned but are not yet wired:

| Status | Feature |
|---|---|
| **Done** | `DragData` MIME map, `DragEffect` / `DropEffect` enums, `NodeDrag` state in `ActiveDragType`, `accept_drop` / `set_drop_effect` / `get_drag_types` / `get_drag_data` on `CallbackInfo`. |
| **Not done** | `DragEnter` / `DragOver` / `DragLeave` / `Drop` event *generation* by the event loop. The filters exist; the loop does not synthesize the events. The `Drop` handler will not fire today. |
| **Not done** | Visual drag feedback: GPU-transform the source node to follow the cursor, and an opacity dim for the source. |
| **Not done** | CSS pseudo-classes `:dragging`, `:drag-over`, `:drag-over-invalid` for styling source and target during a drag. |

Code that follows the API described above will start working as the missing event-generation lands without needing a rewrite.

## Auto-scroll during drag

`AutoScrollDirection` (`core/src/drag.rs`) is set on `TextSelectionDrag` when the cursor approaches the edge of a scroll container during a selection drag. The reserved `DRAG_AUTOSCROLL_TIMER_ID` (see [timers](timers.md#timerid--the-handle)) drives the auto-scroll. For `NodeDrag`, the same wiring is planned but not yet active — manually scroll from your `Drag` callback if you need it today.

## File drops from the OS

OS-level file drops work through `FileDropDrag`. The OS shell hands the framework a list of paths; the gesture manager creates a drag context with `ActiveDragType::FileDrop(FileDropDrag { files, ... })`. Your `DroppedFile` callback reads the file list:

```rust,ignore
extern "C" fn on_dropped_file(_: RefAny, info: CallbackInfo) -> Update {
    if let Some(path) = info.get_dragged_file() {
        // ... open the file ...
    }
    Update::RefreshDom
}
```

`info.get_dragged_file()` returns the first dropped file path. Multi-file drops are exposed via the `FileDropDrag::files` field on the drag context.
