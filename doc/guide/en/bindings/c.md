# C Bindings

> **WIP** — `azul.h` is auto-generated from `api.json`. The header is regenerated on every release; the function and type names are stable for a given `apiversion` value but new symbols land between releases. Pin to a specific commit until your project no longer changes shape.

The C binding consists of two files: `azul.h` (the header) and a shared library (`libazul.so`, `libazul.dylib`, or `azul.dll`). Both are produced by the [code generation pipeline](../code-generation.md). For the program itself, see [Hello, World — C](../hello-world/c.md).

## Get the artifacts

From a checkout of the repository:

```sh
cargo run --release -p azul-doc -- codegen all
cargo build -p azul-dll --release --no-default-features --features build-dll
```

After both commands:

| file | location |
|---|---|
| `azul.h` | `target/codegen/azul.h` |
| `libazul.so` / `libazul.dylib` / `azul.dll` | `target/release/` |
| `azul.dll.lib` (Windows MSVC import lib) | `target/release/` |

For a stable release archive, download the platform tarball from `azul.rs/release/<version>/` — it contains the header, the dylib, and the matching `LICENSE-<PLATFORM>.txt` produced by `doc/src/dllgen/license.rs`.

## Linux — GCC

```sh
gcc -I. hello-world.c -L. -lazul -Wl,-rpath,'$ORIGIN' -o hello-world
./hello-world
```

`-Wl,-rpath,'$ORIGIN'` makes the binary look for `libazul.so` in the same directory as itself, no `LD_LIBRARY_PATH` required. The single-quoted `$ORIGIN` is consumed by the linker, not the shell.

## macOS — Clang

```sh
clang -I. hello-world.c -L. -lazul -o hello-world
./hello-world
```

The release dylib already has its install name set to `@executable_path/libazul.dylib`, so a sibling `libazul.dylib` is found at run time without `DYLD_LIBRARY_PATH`.

If you built the dylib yourself and skipped the install-name fixup, run:

```sh
install_name_tool -id @executable_path/libazul.dylib libazul.dylib
install_name_tool -change /full/path/libazul.dylib @executable_path/libazul.dylib hello-world
```

## Windows — MinGW

```sh
gcc -I. hello-world.c -L. -lazul -o hello-world.exe
hello-world.exe
```

Place `azul.dll` next to `hello-world.exe`. Windows resolves DLLs relative to the executable directory by default; no rpath equivalent is needed.

## Windows — MSVC

Link against the import library `azul.dll.lib`, not `azul.lib` (which is the static archive):

```bat
cl /I. hello-world.c azul.dll.lib /Fehello-world.exe
hello-world.exe
```

## Static linking

For a single-binary build, link against `libazul.a` (Linux/macOS) or `azul.lib` (Windows). The static archive embeds the entire framework — expect a binary in the 30–60 MB range.

```sh
gcc -I. hello-world.c -L. -lazul -lpthread -ldl -lm -o hello-world
```

The Linux static archive does not pull in the C++ standard library or platform GUI libraries; you must link them yourself. Required flags vary by platform:

| platform | extra link flags |
|---|---|
| Linux x86_64 | `-lpthread -ldl -lm -lfontconfig -lX11 -lxkbcommon -lEGL -lGL` |
| macOS | `-framework AppKit -framework CoreFoundation -framework OpenGL` |
| Windows | `gdi32.lib user32.lib opengl32.lib comdlg32.lib uxtheme.lib dwmapi.lib` |

The exact list is platform-dependent; consult `dll/Cargo.toml`'s `[target.'cfg(...)'.dependencies]` blocks for the canonical set.

## Discovering the API surface

`azul.h` is generated by `lang_c.rs` from the IR. The conventions are:

- Every type is prefixed with `Az`: `AzApp`, `AzDom`, `AzString`.
- Every constructor is named `Az<Type>_<methodName>`: `AzString_copyFromBytes`, `AzApp_create`.
- Enums have a `<Type>_Tag` companion type and `<Type>_<Variant>` constructors. For tagged unions the payload is read from `value.<Variant>.payload`.
- Heap-owned types provide a `<Type>_delete(&value)` finalizer. Stack types do not.
- Reference-counted types (e.g. `AzRefAny`) have `<Type>_clone(&value)` and `<Type>_delete(&value)`.

The header is single-file and self-contained — no other azul header is required. It declares only the C ABI; the C++ wrapper lives in a separate header (see [C++ Bindings](cpp.md)).

## Example layout

A minimal project tree:

```
my-app/
├── azul.h              ← copy from target/codegen/
├── libazul.so          ← copy from target/release/
└── hello-world.c
```

Compile, run, ship the binary alongside the dylib. The license file from the release archive (`LICENSE-LINUX.txt` etc.) must accompany binary distributions — it lists the third-party licenses that apply to the bundled crates and is generated by `doc/src/dllgen/license.rs` from `cargo-license` output.
