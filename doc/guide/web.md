# AZ_BACKEND=web — Implementation Plan

## 1. Overview

When the Azul DLL detects `AZ_BACKEND=web://[ip]:[port]`, the framework runs as an HTTP server instead of opening a native window. The env var uses URL-style configuration:

```
AZ_BACKEND=web://127.0.0.1:8080       # localhost only, port 8080
AZ_BACKEND=web://0.0.0.0:3000         # all interfaces, port 3000
AZ_BACKEND=web://[::1]:8080           # IPv6 loopback
AZ_BACKEND=web://0.0.0.0:443?tls=cert.pem  # future: TLS support
```

The server:

1. Renders the initial DOM via `layout()` → returns raw HTML with `<link rel="preload">` hints for all callback WASMs (instant first paint, browser begins parallel-fetching assets immediately)
2. At startup, **pre-transpiles all discovered user callback functions** from x86-64 → WASM
3. Streams `azul-mini.wasm` (the framework core, containing a `HeadlessWindow` equivalent) + per-callback `.wasm` files to the browser
4. After hydration, **all interaction is fully client-side** — zero server round-trips

The key architectural insight: the browser DOM is treated as a **dumb render target**. Azul maintains a complete duplicate of the layout tree in WASM linear memory, performs its own hit-testing on `mousemove`/`mousedown`, and dispatches callbacks internally. The browser never needs to register per-element event listeners — only two global handlers on `<body>`. This mirrors how the native `HeadlessWindow` works, except the final output is DOM mutation calls (via `web_sys`) instead of display list → pixel rendering → swap.

## 3. The Lifting Pipeline (x86-64 → WASM)

### 3.1 Why remill

**Remill** (Trail of Bits) is a library that lifts individual x86-64 instructions to LLVM IR. It produces basic-block-level IR wrapped in a `__remill_basic_block(State&, addr_t, Memory*)` function where `State` is the full x86 register file.

**Key properties relevant to us:**
- Library, not a standalone tool — can be linked into the Azul DLL directly
- Handles x86-64, AArch64 (future: ARM server support)
- Lifts instruction semantics accurately, including integer, SSE, AVX
- Output is standard LLVM IR → can be fed directly to LLVM's wasm32 backend
- C++ library, but we can wrap it via a thin C API or call from `cc` crate build script

**McSema** (also Trail of Bits) builds on remill to lift *entire programs*. We do NOT need McSema — we only lift individual, known-signature functions.

**Why not rellume?** Rellume produces higher-quality IR but is LGPL-licensed, which conflicts with Azul's LGPL + static linking exception model. Remill is Apache-2.0 — no licensing concerns.

### 3.2 The lifting flow for a single callback

```
extern "C" fn on_click(data: AzRefAny, info: AzCallbackInfo) -> AzUpdate

1. dlsym("on_click") → fn_addr = 0x7f1234000
   dladdr(fn_addr)   → fn_size via next symbol or DWARF

2. Read bytes: binary[fn_addr..fn_addr+fn_size]

3. remill::TraceLifter::Lift(fn_addr, fn_size)
   → produces LLVM IR function:
     define %Memory* @lifted_on_click(%State* %state, i64 %pc, %Memory* %mem)
   (Apache-2.0 licensed — clean for Azul's LGPL + static linking exception)

4. Signature recovery (we know the C ABI):
   - System V AMD64: arg0 = %rdi (AzRefAny), arg1 = %rsi (AzCallbackInfo)
   - Return value in %rax (AzUpdate = i32 enum)
   - Extract state.gpr.rdi → wasm param $0
   - Extract state.gpr.rsi → wasm param $1
   - state.gpr.rax after execution → wasm return

5. Patch calls to Azul C API:
   - call @AzRefAny_clone  → (import "azul-mini" "RefAny_clone")
   - call @AzDom_addChild  → (import "azul-mini" "Dom_addChild")
   - call @MyDataModel_downcastMut → inline (it's just a pointer cast + refcount)
   
6. Patch unsupported operations:
   - call to anything involving syscall/file/network
     → replace with panic stub, record for startup warning report

7. LLVM optimization passes:
   - mem2reg, SROA (eliminates most State struct accesses)
   - SimplifyCFG, InstCombine, GVN
   - DCE (removes unused register computations — flags, etc.)
   Result: for `counter += 1`, the IR collapses to ~5 instructions

8. llc -mtriple=wasm32-unknown-unknown -O3 -filetype=obj → .o
   wasm-ld --no-entry --export-all → callback_on_click.wasm
   
   Typical output: 60-150 bytes for simple callbacks
```

### 3.3 Integrating remill into the Azul build

Remill is a C++ library that depends on LLVM. Integration options:

**Option A — Link remill as a C++ dependency (recommended for Phase 1):**
```
azul-dll/
  Cargo.toml          # links to remill via cc crate or cmake
  build.rs            # builds remill from source or finds system install
  src/
    web_backend/
      lifter.rs       # FFI wrapper around remill C++ API
      patcher.rs      # IR patching logic (replace AzXxx calls → imports)
      compiler.rs     # LLVM wasm32 backend invocation
```

The Azul DLL already links against system libraries; adding LLVM/remill as a build-time dependency is feasible. The LLVM dependency is only needed when `AZ_BACKEND=web` feature is enabled.

**Option B — Separate `azul-transpiler` binary:**
A standalone tool that reads the user binary and produces `.wasm` files. The web server invokes it at startup. Simpler build but adds process overhead.

**Option C — Use LLVM C API directly (skip remill):**
Since we know the exact function signatures and the callbacks are compiled from a known toolchain, we could use `iced-x86` for disassembly and write a custom lifter that only handles the subset of x86 instructions that Rust/C compilers actually emit for simple callback functions. Much lighter dependency but far more work and less robust.

**Recommendation:** Start with Option A. Use remill as a library with the LLVM C API for the wasm backend. Gate the entire dependency behind `#[cfg(feature = "web")]`.

### 3.4 Discovering and classifying functions

The key input is `api.json`, which describes every C API function in the Azul DLL. At startup, classify each symbol:

```rust
#[derive(Debug)]
enum FnClass {
    /// Framework fn needed in WASM (AzDom_addChild, AzRefAny_clone, ...)
    /// → lift and include in azul-mini.wasm
    Framework,
    /// The server entry point — NOT included in azul-mini.wasm, but NOT a no-op.
    /// AzApp_run is the function that detects AZ_BACKEND=web://, starts the
    /// server, orchestrates lifting, and serves requests. On desktop it runs
    /// the native event loop. It simply doesn't belong in the browser WASM.
    ServerEntryPoint,
    /// Display list / GPU path fns (AzDisplayList_*, AzGl_*, ...)
    /// → replace with setStyle(nodeId, prop, value) emitters via web_sys
    ReplaceWithDomPatcher,
    /// User callback signature: fn(AzRefAny, AzCallbackInfo) → AzUpdate
    /// → discovered from DOM tree, lifted into separate .wasm files
    UserCallback,
}

fn classify_from_api_json(api: &ApiJson) -> HashMap<String, FnClass> {
    let mut map = HashMap::new();
    for func in &api.functions {
        match func.name.as_str() {
            // AzApp_run is the orchestrator — it runs the server on web,
            // the native event loop on desktop. Excluded from azul-mini.wasm.
            // AzApp_create is fine — it just creates the logical app, nothing visual.
            "AzApp_run"
                => map.insert(func.name.clone(), FnClass::ServerEntryPoint),
            n if n.starts_with("AzDisplayList_") || n.starts_with("AzGl_")
                => map.insert(func.name.clone(), FnClass::ReplaceWithDomPatcher),
            _ => map.insert(func.name.clone(), FnClass::Framework),
        };
    }
    map
}
```

User callbacks are discovered from **all configured routes** (not just the initial DOM):
```rust
fn discover_all_user_callbacks(
    app_config: &AzAppConfig,
    data: &AzRefAny,
) -> Vec<CallbackInfo> {
    let mut callbacks = Vec::new();
    // Iterate all routes → call each layout fn → discover callbacks in each DOM
    for route in &app_config.routes {
        let info = LayoutCallbackInfo::for_route(&route.match_pattern);
        let dom = (route.layout_fn)(data.clone(), info);
        for node in dom.iter_recursive() {
            for (event, cb) in node.callbacks() {
                let fn_ptr = cb.fn_ptr as *const ();
                let name = dladdr_name(fn_ptr);
                let size = find_fn_size(fn_ptr);
                callbacks.push(CallbackInfo { name, fn_ptr, size, event });
            }
        }
        // The layout fn itself is also a callback that needs lifting
        callbacks.push(CallbackInfo::from_layout_fn(route.layout_fn));
    }
    callbacks.dedup_by_key(|c| c.fn_ptr);
    callbacks
}
```

## 4. azul-mini.wasm — Generated at Startup from the Native DLL

### 4.1 Core concept

**There is no separate `azul-mini` crate. There is no wasm32 build step.**

`azul-mini.wasm` is generated at server startup by lifting the framework functions directly out of the running native binary (`azul.so` / `azul.dll`). The process:

1. Read `api.json` → get the list of every Azul C API function
2. For each framework function: `dlsym` → locate in binary → lift via remill → LLVM IR
3. **Exclude** `ServerEntryPoint` fns (`AzApp_run`, etc.) — they don't belong in browser WASM
4. Replace display list / GPU rendering path with `setStyle()` emitters (cascade output → inline styles via web_sys)
5. Link all lifted framework functions into a single LLVM module
6. Compile that module → `azul-mini.wasm`

This means the build process is **completely unchanged** — `cargo build --release` produces a regular native DLL. All the WASM generation complexity is deferred to server startup.

### 4.2 Rendering model: Azul cascades, browser renders

```
Native:  layout() → cascade → solve → display list → GPU render → swap
Web:     layout() → cascade → solve → diff computed styles + DOM structure
                                    → emit element.style.X = Y per node
                                    → browser renders with those inline styles
                                    → update internal layout cache (for hit-testing)
```

Azul runs its **own CSS cascade in WASM** — the browser's cascade is never used. This means:

- Azul's custom `@` rules and CSS components work unchanged
- Cascading behavior is identical on Firefox, Chrome, Safari (decoupled from browser CSS engine)
- The cascade stays entirely in WASM (no JS boundary crossing during computation)

After cascade + layout, Azul emits **direct inline style mutations** per node:

```
Elements are identified by stable IDs:   <div id="az_403">...</div>

On re-layout, Azul diffs computed styles and emits only changes:
  document.getElementById('az_403').style.fontWeight = 'bold';
  document.getElementById('az_403').style.color = '#333';
  document.getElementById('az_817').style.display = 'none';
```

The browser receives fully-resolved inline styles — it doesn't need to do any cascade resolution, selector matching, or specificity computation. It just applies the styles and paints.

On the web target, the **browser handles only:**
- Applying inline styles that Azul sets via `element.style.*`
- Text rendering and font rasterization
- Scrolling (native scroll behavior)
- Painting and compositing

**Azul handles in WASM:**
- CSS cascade (including custom `@` rules, CSS components, variables)
- Layout solving (reftested against Chrome)
- DOM diffing → structural patches (add/remove/move nodes)
- Style diffing → inline style patches (only changed properties)
- Hit-testing against internal layout cache
- Callback dispatch

### 4.3 What gets lifted into azul-mini.wasm

| Category | Example functions | Treatment |
|----------|-------------------|-----------|
| DOM construction | `AzDom_createDiv`, `AzDom_addChild`, `AzDom_style` | Lift as-is |
| Layout solver | `AzLayout_solve`, `AzLayout_*` | Lift as-is (identical to native) |
| CSS cascade | `AzCss_cascade`, `AzCssProperty_*` | Lift as-is (Azul's cascade, not the browser's — supports custom `@` rules, CSS components) |
| DOM diffing | `AzDom_diff` | Lift as-is |
| Hit-testing | `AzHitTest_*` | Lift as-is (same algorithm as native) |
| RefAny | `AzRefAny_clone`, `AzRefAny_downcastRef/Mut` | Lift as-is |
| Widgets | `AzButton_create`, `AzTextInput_*`, etc. | Lift as-is |
| Routing | `AzCallbackInfo_switchRoute`, `AzCallbackInfo_setRouteOpts` | Lift as-is (see §5.4) |
| App lifecycle | `AzApp_create`, `AzApp_delete` | Lift as-is (creates logical app, nothing visual) |
| App entry point | `AzApp_run` | **Excluded** — orchestrates server/desktop event loop, not browser |
| GPU / Display list | `AzDisplayList_*`, `AzGl_*`, `AzRenderer_*` | Replace with `setStyle(nodeId, prop, value)` emitters via web_sys |
| Window management | `AzWindow_*` | Replace with browser-dimension shims |
| File I/O, networking | `AzFile_*`, `AzTcp_*` | Replace with panic stubs (see §6.3); long-term: browser-native shims |

Each user callback `.wasm` file is a tiny module (~60-150 bytes) that imports the framework functions it needs from `azul-mini.wasm`. The JS loader instantiates `azul-mini.wasm` first, then passes its exports as the import object for each callback module.

### 4.5 Why Azul owns the cascade (not the browser)

The browser's CSS engine is **bypassed for cascading**. Azul computes all styles in WASM and applies them as inline styles (`element.style.*`). This gives us:

1. **Custom `@` rules and CSS components** — Azul's CSS extensions work unchanged because the browser never parses our stylesheets
2. **Cross-browser consistency** — cascade results are identical on Firefox/Chrome/Safari because Azul computes them, not the browser's engine
3. **Performance** — cascade stays in WASM (no JS↔browser style system round-trips). Only the final computed style *changes* cross the boundary
4. **Hit-testing** — Azul's internal layout cache (from the same cascade + solve pass) enables `hit_test(x, y)` without querying the DOM
5. **Resolution independence** — Azul controls how CSS values resolve relative to the viewport, independent of browser DPI/zoom behavior

**Resource fetching:** Fonts need raw TTF/OTF bytes fetched into WASM (for `allsorts`). Images are decoded by the browser natively — azul-mini.wasm ships **no image decoders** (no png/jpeg/webp crates). Both are prefetched via `<link rel="preload">` in the initial HTML.

**Optimization opportunity:** The patcher can aggressively strip azul-mini.wasm — eliminate debug assertions, panic formatting strings, unused code paths, image decoder code that exists in the native DLL but isn't needed on web. Working at the LLVM IR level before wasm compilation gives fine-grained control over what ends up in the final binary.

### 4.6 Size estimate

The framework has ~200 C API functions. Many are thin wrappers. After lifting + LLVM optimization + wasm-opt:
- Estimated uncompressed: **80-120KB**
- Estimated compressed (brotli): **25-45KB**
- This is a one-time cost, aggressively cached by the browser (content-hashed URL, immutable)

## 5. The Web Server

### 5.1 AzApp_run as the orchestrator

`AzApp_run` is the **single entry point** for both desktop and web. It detects `AZ_BACKEND` and branches:

```rust
// ── User code (identical for desktop and web) ──

int main() {
    MyDataModel model = { .counter = 5 };
    AzRefAny data = MyDataModel_upcast(model);

    AzAppConfig config = AzAppConfig_create();
    
    // Register routes: each route maps to a layout callback fn
    AzAppConfig_addRoute(&config, "/",        layout_home,    data);
    AzAppConfig_addRoute(&config, "/about",   layout_about,   data);
    AzAppConfig_addRoute(&config, "/user/:id", layout_user,   data);
    
    AzApp app = AzApp_create(data, config);
    AzApp_run(&app);  // ← this is the magic function
    AzApp_delete(&app);
    return 0;
}

// layout callbacks receive route info via LayoutCallbackInfo
AzDom layout_user(AzRefAny data, AzLayoutCallbackInfo info) {
    // info.route_match.pattern = "/user/:id"
    // info.route_match.params["id"] = "42"
    AzString user_id = AzLayoutCallbackInfo_getRouteParam(&info, "id");
    // ... build DOM for this user
}
```

Inside `AzApp_run`:

```rust
// ── Inside Azul framework ──

pub fn AzApp_run(app: &AzApp) {
    let backend = std::env::var("AZ_BACKEND").unwrap_or_default();
    
    if let Some(bind_addr) = parse_web_url(&backend) {
        // ═══ WEB PATH ═══
        web_server_startup(app, bind_addr);
    } else {
        // ═══ DESKTOP PATH (unchanged) ═══
        let window = create_native_window(app);
        native_event_loop(app, window);
    }
}

fn web_server_startup(app: &AzApp, bind_addr: SocketAddr) {
    let api = load_embedded_api(); // decompress brotli include_bytes!("api.json.br")
    let lifter = AzLifter::new();
    
    // ── Phase A: Generate azul-mini.wasm from framework fns ──
    let fn_classes = classify_from_api_json(&api);
    let mut mini_module = lifter.create_module("azul-mini");
    
    for (fn_name, class) in &fn_classes {
        match class {
            FnClass::Framework => {
                let addr = dlsym_or_panic(fn_name);
                let ir = lifter.lift_function(addr);
                mini_module.add_function(fn_name, ir);
            }
            FnClass::ServerEntryPoint => { /* exclude from WASM */ }
            FnClass::ReplaceWithDomPatcher => {
                mini_module.add_set_style_emitter(fn_name);
            }
            FnClass::UserCallback => { /* handled in Phase B */ }
        }
    }
    
    mini_module.optimize();
    let mini_wasm = mini_module.compile_to_wasm();
    let mini_hash = content_hash(&mini_wasm);
    
    // ── Phase B: Discover + lift ALL user callbacks across ALL routes ──
    let user_callbacks = discover_all_user_callbacks(&app.config, &app.data);
    
    let cb_wasms: HashMap<String, (Vec<u8>, ContentHash)> = user_callbacks.iter()
        .map(|cb| {
            let ir = lifter.lift_function(cb.fn_ptr);
            let patched = lifter.relink_against_mini(&ir, &fn_classes);
            let wasm = lifter.compile_to_wasm_module(&patched);
            let hash = content_hash(&wasm);
            (cb.name.clone(), (wasm, hash))
        })
        .collect();
    
    // ── Phase C: Pre-render each route's initial HTML ──
    let mut route_html: HashMap<String, String> = HashMap::new();
    for route in &app.config.routes {
        let info = LayoutCallbackInfo::for_route(&route.match_pattern);
        let dom = (route.layout_fn)(app.data.clone(), info);
        let preloads = generate_preload_hints(&mini_hash, &cb_wasms);
        let state = dom.serialize_layout_state();
        route_html.insert(route.match_pattern.clone(), 
            dom.render_to_html_with_head(&preloads, &state));
    }
    
    // ── Phase D: Start HTTP server ──
    start_web_server(bind_addr, route_html, mini_wasm, cb_wasms, app);
}
```

### 5.2 HTTP endpoints

| Endpoint | Response | Cache |
|----------|----------|-------|
| `GET /` | Pre-rendered HTML for `/` route with `<link rel="preload">` hints | no-cache |
| `GET /about` | Pre-rendered HTML for `/about` route | no-cache |
| `GET /user/:id` | Pre-rendered HTML for `/user/:id` route (params in state) | no-cache |
| `GET /az/mini.{hash}.wasm` | Framework core (lifted HeadlessWindow) | immutable, forever |
| `GET /az/cb/{name}.{hash}.wasm` | Individual callback WASM, content-hashed | immutable, forever |
| `GET /az/loader.js` | Tiny bootstrap script (~300B) | immutable |

All `/az/` assets use content-hashed filenames. Route endpoints (`/`, `/about`, `/user/42`) serve the pre-rendered HTML for that route's layout callback. The browser's native parallel fetch, triggered by the `<link rel="preload">` tags, ensures that by the time `loader.js` executes, all WASMs are already in the HTTP cache.

### 5.3 Routing — a cross-platform abstraction

Routing is a **first-class framework concept** that works identically on desktop and web. The user registers routes in `AppConfig`, each mapping a URL pattern to a layout callback:

```c
// C API additions
void AzAppConfig_addRoute(
    AzAppConfig* config,
    AzString pattern,          // "/user/:id", "/about", "/"
    AzLayoutCallback layout_fn, // the layout() for this "screen"
    AzRefAny data
);

// Inside a callback, switch to a different route:
AzUpdate on_click_about(AzRefAny data, AzCallbackInfo info) {
    AzCallbackInfo_switchRoute(&info, "/about", AzRouteOpts_default());
    return AzUpdate_RefreshDom;
}

// Or with parameters:
AzUpdate on_click_user(AzRefAny data, AzCallbackInfo info) {
    AzRouteOpts opts = AzRouteOpts_default();
    AzRouteOpts_setParam(&opts, "id", "42");
    AzCallbackInfo_switchRoute(&info, "/user/:id", opts);
    return AzUpdate_RefreshDom;
}
```

**What `switchRoute` does per target:**

| Operation | Desktop | Web |
|-----------|---------|-----|
| `switchRoute("/about", opts)` | Swap active layout fn pointer on the window, update `RouteArgs` in `LayoutCallbackInfo`, trigger re-layout | Call `history.pushState()`, swap active layout fn in WASM dispatch table, trigger re-layout |
| `getRouteParam("id")` | Read from `LayoutCallbackInfo.route_match.params` | Same — stored in WASM linear memory |
| Browser back/forward | N/A (no browser) | `onpopstate` → WASM `switchRoute` to the previous pattern |

**LayoutCallbackInfo extension:**

```c
// New fields in LayoutCallbackInfo
typedef struct {
    AzString pattern;           // "/user/:id"
    AzStringPairVec params;     // [("id", "42")]
} AzRouteMatch;

// Accessible in layout():
AzDom layout_user(AzRefAny data, AzLayoutCallbackInfo info) {
    AzString id = AzLayoutCallbackInfo_getRouteParam(&info, "id");
    // ... build DOM based on route params
}
```

This means the user writes route-aware code exactly once — it works on desktop (switching "screens" by swapping the layout fn) and on web (URL-based navigation with `history.pushState`).

### 5.4 The loader script (azul-loader.js)

```javascript
// ~500 bytes minified — the ONLY JavaScript in the entire system
const nodes = {}; // az_id → DOM element cache

(async () => {
    // Assets are already in browser cache from <link rel="preload">
    // These fetches resolve instantly from cache
    
    // 1. Instantiate azul-mini.wasm
    const mini = await WebAssembly.instantiateStreaming(
        fetch('/az/mini.wasm'), {
            web_sys: {
                // DOM structure mutations
                createElement: (tag) => document.createElement(tag),
                setId: (nodeId) => { const el = ...; el.id = `az_${nodeId}`; nodes[nodeId] = el; },
                setText: (nodeId, text) => nodes[nodeId].textContent = text,
                appendChild: (parentId, childId) => nodes[parentId].appendChild(nodes[childId]),
                removeChild: (parentId, childId) => nodes[parentId].removeChild(nodes[childId]),
                insertBefore: (parentId, childId, refId) =>
                    nodes[parentId].insertBefore(nodes[childId], nodes[refId]),
                // Inline style mutations (output of Azul's CSS cascade)
                setStyle: (nodeId, prop, value) => nodes[nodeId].style[prop] = value,
                removeStyle: (nodeId, prop) => nodes[nodeId].style[prop] = '',
                // Resource fetching
                fetchFontIntoWasm: (url, ptr, len) => { /* fetch TTF/OTF → copy bytes into WASM memory */ },
                decodeImage: (url, cb) => { /* browser decodes natively, returns dimensions to WASM */ },
                // Routing
                historyPushState: (url) => history.pushState(null, '', url),
            }
        }
    );
    
    // 2. Hydrate: reconstruct layout tree in WASM from server state
    mini.instance.exports.hydrate_from_dom();
    
    // 3. Load callback WASMs (already in cache from preload)
    const cbManifest = mini.instance.exports.get_callback_manifest();
    for (const { name, hash } of cbManifest) {
        const mod = await WebAssembly.instantiateStreaming(
            fetch(`/az/cb/${name}.${hash}.wasm`),
            { "azul-mini": mini.instance.exports }
        );
        mini.instance.exports.register_callback(name, mod.instance.exports.call);
    }
    
    // 4. Register ONLY global event listeners — Azul does its own hit-testing
    document.body.addEventListener('mousemove', (e) => {
        mini.instance.exports.on_mouse_move(e.clientX, e.clientY);
    });
    document.body.addEventListener('mousedown', (e) => {
        mini.instance.exports.on_mouse_down(e.clientX, e.clientY, e.button);
    });
    document.body.addEventListener('mouseup', (e) => {
        mini.instance.exports.on_mouse_up(e.clientX, e.clientY, e.button);
    });
    document.body.addEventListener('keydown', (e) => {
        mini.instance.exports.on_key_down(e.keyCode, e.shiftKey, e.ctrlKey, e.altKey);
    });
    window.addEventListener('resize', () => {
        mini.instance.exports.on_resize(window.innerWidth, window.innerHeight);
    });
    document.addEventListener('scroll', (e) => {
        mini.instance.exports.on_scroll(window.scrollX, window.scrollY);
    }, { passive: true });
    
    // Routing: browser back/forward → WASM switches to previous route
    window.addEventListener('popstate', (e) => {
        mini.instance.exports.on_route_change(location.pathname);
    });
})();
```

Note: there are **no per-element event listeners**. The `mousemove` handler fires into WASM, which does `hit_test(x, y)` against its internal layout tree to determine hover state. `mousedown` does the same, then dispatches to the appropriate callback. This is identical to how native Azul handles input — the OS sends raw mouse events, Azul does hit-testing internally.

The `web_sys` imports to azul-mini.wasm also include `history_push_state(url)` so that when a callback calls `AzCallbackInfo_switchRoute`, the WASM can push the new URL to the browser's history.

## 6. The IR Patching Strategy

This is the most critical and novel part of the implementation. When remill lifts a user callback, the resulting LLVM IR contains calls to Azul C API functions (like `AzRefAny_clone`, `AzDom_addChild`, etc.) as direct function calls or indirect calls through the PLT.

### 6.1 Identifying Azul API calls in lifted IR

After lifting, scan the LLVM module for:
- Direct calls to symbols matching `Az*` or known Azul function names
- Indirect calls through known PLT entries (resolvable via relocation info)

The complete list of Azul C API functions is in `api.json`. Generate a mapping table:

```rust
// Auto-generated from api.json
static AZUL_API_TO_WASM_IMPORT: &[(&str, &str)] = &[
    ("AzRefAny_clone",       "RefAny_clone"),
    ("AzRefAny_downcastRef", "RefAny_downcastRef"),
    ("AzRefAny_downcastMut", "RefAny_downcastMut"),
    ("AzDom_createDiv",      "Dom_createDiv"),
    ("AzDom_createText",     "Dom_createText"),
    ("AzDom_addChild",       "Dom_addChild"),
    ("AzDom_style",          "Dom_style"),
    ("AzButton_create",      "Button_create"),
    ("AzButton_setOnClick",  "Button_setOnClick"),
    // ... ~200 more entries from api.json
];
```

### 6.2 Replacing calls in LLVM IR

For each identified call:
```
; Before patching:
  %result = call i64 @AzRefAny_clone(%struct.AzRefAny* %data)

; After patching:
  %result = call i64 @import_RefAny_clone(%struct.AzRefAny* %data)
  ; where @import_RefAny_clone is declared as:
  declare i64 @import_RefAny_clone(%struct.AzRefAny*) #wasm_import("azul-mini", "RefAny_clone")
```

At the WASM level, this becomes:
```wasm
(import "azul-mini" "RefAny_clone" (func $RefAny_clone (param i32) (result i32)))
```

### 6.3 Handling unsupported operations

For callbacks that call `libc` functions, syscalls, or platform APIs:

**Category A — Pure computation (safe to run in WASM):**
`memcpy`, `memset`, `strlen`, basic math → inline or provide via a tiny libc shim in azul-mini.wasm

**Category B — Unsupported (panic with clear message):**
`open`, `read`, `write`, `connect`, `send`, `recv`, `fork`, `exec`, signal handlers, etc.

Replace with a panic stub:
```wasm
(import "azul-panic" "unsupported_fn" (func $panic (param i32) (result i32)))
;; param = pointer to error message string in WASM linear memory
```

The patcher **detects these at startup** and the server prints a clear report:

```
[azul-web] Startup complete. Lifted 8 user callbacks, 194 framework fns.
[azul-web] WARNING: 2 code paths contain unsupported operations:
  • on_save_file() calls libc::open() at offset 0x4a2
    → will panic in browser with: "AzFile_open is not yet supported on web target"
  • on_connect() calls libc::connect() at offset 0x891
    → will panic in browser with: "AzTcp_connect is not yet supported on web target"
[azul-web] All other callbacks are fully client-side.
[azul-web] Listening on http://127.0.0.1:8080
```

The WASM does **not** silently fail or unexpectedly crash — the developer knows at startup exactly which code paths will panic.

**Long-term roadmap for Category B functions:**
These panics will be replaced one-by-one with browser-native equivalents patched into azul-mini.wasm:

| Native operation | Future web replacement |
|-----------------|----------------------|
| `AzFile_open` (read) | `<input type="file">` picker dialog |
| `AzFile_write` | Download via `Blob` + `URL.createObjectURL` |
| `AzTcp_connect` | `WebSocket` or `fetch()` proxy |
| `AzTimer_*` | `setTimeout` / `requestAnimationFrame` |
| `AzThread_*` | `Web Worker` |
| `AzClipboard_*` | `navigator.clipboard` API |

Each replacement is a shim patched into the lifted IR — the user's callback code stays unchanged. The goal is to drive the startup warning list down to zero over time.

## 8. Key Technical Risks

### Risk 1: remill IR quality for complex callbacks
**Concern:** remill's lifted IR models the full x86 register file as a State struct. For simple callbacks (`counter += 1`), LLVM's optimization passes eliminate most of this. For complex callbacks with loops, branches, and many local variables, the IR may not optimize as cleanly.

**Mitigation:** 
- Run aggressive LLVM passes: `-O3` with SROA, mem2reg, GVN, SimplifyCFG
- For callbacks where lifted WASM exceeds a size threshold (say 4KB), fall back to server round-trip mode
- If IR quality is still poor after optimization, consider writing targeted peephole passes for common Rust/C callback patterns

### Risk 2: Calling convention recovery
**Concern:** Mapping x86-64 registers to WASM function parameters requires knowing the exact calling convention. System V AMD64 is well-defined, but LTO/inlining may alter function boundaries.

**Mitigation:**
- Compile with `-C lto=thin` or `fat` but preserve symbol boundaries
- Use DWARF debug info for function boundary detection (compile with `-g` even in release for the web target)
- The user-facing C ABI functions are `#[no_mangle] extern "C"` — these are guaranteed to follow the platform ABI

### Risk 3: Indirect calls
**Concern:** Some Azul internal functions use indirect calls (function pointers in structs). The lifter can't resolve these statically.

**Resolution:** This is largely a non-issue. All Azul callbacks are required to be `extern "C"`, so every indirect call target is a named symbol discoverable via `dlsym`. At startup, the patcher can resolve all indirect call sites by scanning the binary's symbol table and matching function pointer values to their symbol names. The lifted WASM then uses direct named calls instead of indirect ones.

### Risk 4: LLVM dependency size
**Concern:** LLVM is often cited as a massive dependency (~500MB+ build artifacts). Including it in the Azul DLL could bloat the binary.

**Resolution:** This is overstated. Wasmer ships an LLVM-based compiler without anywhere near 50MB+ overhead — they link only the subset of LLVM needed (the wasm backend + optimization passes, no clang, no other targets). The same approach applies here:
- Link only `LLVMCore`, `LLVMWebAssemblyCodeGen`, `LLVMWebAssemblyAsmParser`, `LLVMPasses` — skip all other backends and tools
- Gate behind `feature = "web"` so native-only builds don't pull it in
- Use a pre-built LLVM from system packages (link dynamically) to avoid build-time cost
- Long-term: explore `cranelift` as a lighter alternative (it supports wasm32 output natively)

### Risk 5: Font / image fetching and azul-mini.wasm size
**Concern:** `allsorts` + `azul-layout` + all framework fns could make azul-mini.wasm larger than estimated.

**Resolution:**
- **Fonts:** Only the raw font data (TTF/OTF bytes) needs to be fetched into WASM. `allsorts` handles all shaping — no browser font APIs needed. Prefetch via `<link rel="preload">`.
- **Images:** The browser fetches AND decodes images natively (`fetch()` + `createImageBitmap()` or `<img>`). azul-mini.wasm does **not** ship image decoders (no png/jpeg/webp crate). Only decoded pixel data is passed into WASM if needed for layout sizing.
- **Size stripping:** The patcher can aggressively strip at the LLVM IR level — eliminate panic formatting strings, debug assertions, unused `Display`/`Debug` impls, `core::fmt` machinery, and dead code paths. This is more effective than post-hoc `wasm-opt` because we remove entire unreachable call subtrees.
- If `allsorts` alone adds 200KB+, consider lazy-loading it as a separate `azul-fonts.wasm` module.

### Risk 6: Server startup time
**Concern:** Lifting ~200 framework functions + N user callbacks via remill + LLVM optimization + wasm32 compilation at every server start could take 10-60 seconds. This is the cost of "no build step" — the complexity moves to startup.

**Mitigation:**
- **Cache the generated WASMs on disk.** Hash the input binary + api.json → if unchanged, skip lifting entirely and load cached `.wasm` files. First startup is slow, subsequent starts are instant.
- Parallelize lifting: each function can be lifted independently. Use `rayon` to lift on all cores simultaneously.
- Use LLVM's `-O1` for development, `-O3` only for production (the `?opt=3` query param on the URL).
- Long-term: explore incremental lifting — only re-lift functions whose bytes changed since last startup.

## 9. Alternative Approach: Dual Compilation (Skip Lifting Entirely)

If remill integration proves too complex, there's a simpler (but less elegant) alternative:

**Compile the user code twice:**
1. `cargo build --target x86_64-unknown-linux-gnu` → native server binary
2. `cargo build --target wasm32-unknown-unknown` → user callbacks as WASM

The server binary serves the pre-compiled WASM callbacks. This requires the user to have both targets installed and run two build steps, but eliminates the entire lifting pipeline.

**Pros:** No LLVM/remill dependency, simpler architecture, guaranteed correct WASM output
**Cons:** Requires user toolchain setup, two compilation steps, C users need wasi-sdk, loses the "single binary" elegance

This could serve as a **fallback mode** (`AZ_BACKEND=web-precompiled://127.0.0.1:8080`) while the lifting pipeline matures.

## 12. Success Criteria

The implementation is considered successful when:

1. **Build process is unchanged:** `cargo build --release` produces the same native DLL as today. No wasm32 target, no dual compilation, no new build steps.
2. **Counter app works end-to-end:** Click "Increment" → counter updates in browser, zero server round-trips after initial load
3. **First paint under 50ms:** Raw HTML renders before any WASM is loaded (preload hints ensure assets are already fetching)
4. **Callback WASM files under 200 bytes each:** For typical callbacks (field mutation + return RefreshDom)
5. **azul-mini.wasm under 100KB compressed:** ~200 framework fns lifted from native, optimized, linked into one module
6. **Server startup under 5s (cached: instant):** First startup lifts everything; subsequent starts load from disk cache
7. **Full page interactivity under 200ms:** Time from first byte to all callbacks registered and hit-testing active
8. **krausest benchmark geometric mean under 1.10x:** Competitive with top-tier JS frameworks
9. **Zero per-element addEventListener calls:** Only global `mousemove` + `mousedown` + `keydown` + `resize` + `scroll` on body
10. **Hit-testing accuracy matches native:** Azul's internal layout cache (reftested against Chrome) correctly identifies click targets without per-element listeners
