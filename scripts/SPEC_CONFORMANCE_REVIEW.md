## core/src/dom.rs

- **PARTIAL ‚Äî display-property:844893** ‚Äî The annotation (core/src/dom.rs:1076) claims `FormattingContext` models "a block-level box establishing a new formatting context (BFC)". The core of that claim holds: `Block { establishes_new_context }` (core/src/dom.rs:1079‚Äì1082) correctly models css-display-3's "a block container ‚Ä¶ either establishes a new block formatting context for its contents or continues the one in which it participates", and the production predicate feeding it (`establishes_new_block_formatting_context`, layout/src/solver3/layout_tree.rs:2796‚Äì2852) correctly implements the CSS 2.2 ¬ß9.4.1 trigger list (inline-block/table-cell/table-caption/flow-root, overflow, float, absolute/fixed, plus writing-mode change, replaced elements, and the root), including the css-overflow-3 carve-out that `overflow: clip` does *not* establish a formatting context ("neither visible nor clip nor a combination thereof") via `is_visible_or_clip()`. However, the model deviates from the spec in four concrete ways:
  - **The BFC bit is silently dropped for block containers with only inline-level content.** `determine_formatting_context_for_display` returns `FormattingContext::Inline` for `display:block` when `has_only_inline_children` is true *without ever consulting the BFC predicate* (layout/src/solver3/layout_tree.rs:2890‚Äì2893); the `Inline` variant (core/src/dom.rs:1084) has no place to record BFC establishment. So `<div style="overflow:hidden">text</div>` or a floated block containing only text is modeled as a plain inline FC with no BFC ‚Äî CSS 2.2 ¬ß9.4.1 requires "block boxes with 'overflow' other than 'visible' ‚Ä¶ establish new block formatting contexts" regardless of content, and css-display-3 says such a box has used inner display `flow-root`. The asymmetry is visible in the same match: `display:flow-root` with identical text content keeps its BFC (layout_tree.rs:2886‚Äì2888), `overflow:hidden` loses it.
  - **The `Float`/`OutOfFlow` variants are dead code, so out-of-flow boxes are not actually "modeled here".** `FormattingContext::Float(LayoutFloat)` and `FormattingContext::OutOfFlow(LayoutPosition)` (core/src/dom.rs:1090‚Äì1092) are never constructed anywhere in production ‚Äî floats and abs-pos elements instead go through blockification (layout_tree.rs:1283‚Äì1295) and get `Block { establishes_new_context: true }` from the predicate (layout_tree.rs:2816‚Äì2824). That path is behaviorally spec-correct, but the enum's out-of-flow model is vestigial. The only code constructing those variants is tests/src/layout.rs (1473, 1489, 1524, 1540), which is **not compiled** (tests/src/lib.rs:13‚Äì14 includes `layout-test.rs`, not `layout.rs`) and asserts stale, spec-wrong semantics ‚Äî e.g. tests/src/layout.rs:1503‚Äì1509 claims "Relative positioning establishes a new BFC"; per CSS 2.2 ¬ß9.4.1 it does not (production correctly checks only `is_absolute_or_fixed()`, layout_tree.rs:2817).
  - **Unset overflow spuriously establishes a BFC on the slow cascade path.** When the compact cache is unavailable or the node is in a non-normal styled state, an *unset* overflow resolves to the `MultiValue::Auto` "not set" sentinel (getters.rs macro fallback, layout/src/solver3/getters.rs:669‚Äì670), which is indistinguishable from CSS `overflow:auto` and fails `is_visible_or_clip()` (getters.rs:497‚Äì502) ‚Üí `establishes_new_block_formatting_context` returns true (layout_tree.rs:2806‚Äì2814) for every plain block. The initial value of overflow is `visible` (correctly annotated at css/src/props/layout/overflow.rs:24‚Äì26), which must *not* establish a BFC. The fast path returns `Exact(Visible)` and is correct, so the two paths disagree.
  - **Minor:** the CSS 2.2 ¬ß9.4.1 exemption "(except when that value has been propagated to the viewport)" is not implemented ‚Äî a `<body>` whose overflow is propagated to the viewport should have used value `visible` and not establish a BFC, but layout_tree.rs:2806‚Äì2814 has no propagation check. Additionally, the enum cannot express the css-display-3 (outer, inner) display pair outside the flow family: `inline-flex`/`inline-grid`/`inline-table` map to the same `Flex`/`Grid`/`Table` variants as their block-level forms (layout_tree.rs:2912, 2921, 2924), so "block-level vs. atomic-inline" ‚Äî the very distinction this annotation is about ‚Äî is unrepresentable for non-flow formatting contexts (only `InlineBlock`, core/src/dom.rs:1086, gets that treatment).
The audit is complete. Everything needed is in hand ‚Äî here is the report.

- **CORRECT** ‚Äî `width-calculation:bef810` ‚Äî The paragraph (CSS 2.2 ¬ß8.3 margin definitions) requires margin percentages to resolve against the containing block's width, explicitly including `margin-top`/`margin-bottom`. - **CORRECT** ‚Äî `width-calculation:d78514` ‚Äî The paragraph repeats the same normative rule this annotation claims ("Percentages: refer to width of containing block" for all four margin properties and the `margin` shorthand), and the shared `Margin | Padding` arm at `pixel.rs:570-572` implements it as analyzed above.
