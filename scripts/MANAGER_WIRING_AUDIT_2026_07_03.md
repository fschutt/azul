# Manager wiring audit - layout/src/managers/* vs shell2 backends (2026-07-03)

## 1. Method

21 parallel auditors, one per manager subsystem (gesture, scroll, a11y, file_drop, drag_drop, hover, focus_cursor, text_input, text_edit, clipboard, undo_redo, gamepad, virtual_view, gpu_state, changeset, permission, geolocation, biometric, keyring, sensors, csd), each producing a capability-by-backend wiring matrix (macOS / Windows / X11 / Wayland / Headless) plus ranked gaps. An adversarial verification pass re-derived every critical claim and every claim touching the five user-named scenarios from source, hunting for alternate code paths that would refute them. Verdicts: 36 CONFIRMED, 1 ADJUSTED, 1 REFUTED (see appendix). Audited tree: `/Users/fschutt/Development/azul-mobile`, branch `master`, commit **30162fa9f**. Note: the unmerged branch `fix/shell2-damage-present-wiring` (080a537bf et al.) resolves some Windows CPU-mode hit-test gaps noted below; this report describes master.

Cell legend: OK = every capability row wired; PARTIAL = worst row is partial; DEAD = at least one capability row entirely missing on that backend (the caveat column names it). All file:line citations are preserved verbatim from the auditors.

## 3. The five user-named scenarios

### (a) File dropping
**State:** Genuinely wired end-to-end on all 4 desktop backends — drop-target registration, drag-enter/leave, drop, and path extraction all real (macOS NSDraggingDestination macos/mod.rs:723-741/2599-2626; Windows OLE IDropTarget dnd.rs:58-208; X11 XDND v5 x11/mod.rs:1346-2667; Wayland wl_data_device wayland/events.rs:768-987). X11/Wayland even hit-test the exact drag position.
**Root causes of remaining breakage:** (1) all three file events are targeted at the DOM root (event_determination.rs:654-680), and event_type_to_filters emits only EF::Hover (core/src/events.rs:2446-2448), so a HoveredFile/DroppedFile callback on any non-root node never fires — per-node drop targets are impossible (verifier found it slightly worse than reported: even WindowEventFilter registrations are unreachable). (2) Multi-file drops truncated to paths.first() at all 8 ingress sites; manager stores one Option<AzString> (file_drop.rs:11-13). (3) Headless has zero file-drop wiring — not E2E-testable. (4) macOS/Windows refresh the drag hit test at the stale cached cursor instead of the OS-provided drag location (macos/events.rs:841-844; dnd.rs:153,172,194).
**Fix sketch:** target file events at hover_manager's mouse_target with root fallback; widen manager to StringVec + get_dropped_files; add headless simulate_file_drop; pass draggingLocation/POINTL through.
**Verification:** root-target, multi-file truncation, and headless gaps all CONFIRMED.

### (b) Select-and-drag auto-scroll (incl. pointer outside the window)
**State:** An auto-scroll timer exists (common/event.rs:327-431) but is started ONLY for text-selection drags (core/src/events.rs:3234) — node drag-and-drop and OS file drags never auto-scroll (CONFIRMED; the manager's own auto_scroll_direction plumbing has zero readers). Outside-the-window behavior diverges: macOS keeps delivering mouseDragged with InWindow coords (works); Wayland's implicit grab keeps pointer focus (works); X11 delivers LeaveNotify to the grab window → cursor becomes OutOfWindow → the timer self-terminates (common/event.rs:344-350 + core/window.rs:579-583) and only restarts on the next motion — holding the pointer still past the edge stops scrolling (CONFIRMED, incl. no crossing-mode filter at x11/events.rs:603-606). Windows has the same termination if WM_MOUSELEAVE fires under SetCapture (mod.rs:2908-2932; open question, needs a Windows box). Separately, Windows scrollbar-thumb drag returns before SetCapture (mod.rs:2961-2969 vs :3015) so it freezes at the border (CONFIRMED).
**Root causes:** timer terminates on OutOfWindow instead of using stored coords; StartAutoScrollTimer pushed only from TextSelectionDrag.
**Fix sketch:** treat CursorPosition::OutOfWindow(pos) as valid in auto_scroll_timer_callback (terminate only on button-up); also push StartAutoScrollTimer when is_node_drag_active()/file-hover is true; SetCapture in the Windows scrollbar-click branch.
**Verification:** X11 termination CONFIRMED, text-selection-only CONFIRMED, Windows scrollbar capture CONFIRMED.

### (c) Fake macOS titlebar window drag + HiDPI
**State:** PARTIALLY as advertised. The incremental screen-space-delta design (titlebar.rs:544-575, common/event.rs:3529-3558) genuinely survives DPI changes mid-drag, and macOS stays in points so scale factor itself does not corrupt the math; windowDidChangeBackingProperties handles scale changes (macos/mod.rs:2144-2155).
**Root causes of the bugs that remain (all CONFIRMED):** it is a manual per-event setFrameTopLeftPoint loop, NOT performWindowDragWithEvent (zero hits repo-wide) → no OS-smooth drag, no snap; the Y-flip uses window.screen().frame().size.height and ignores frame().origin (macos/mod.rs:4110-4119, read-back mod.rs:2119-2123) → stored top-down position is wrong in absolute terms on secondary monitors, and when the window crosses to a screen of different height mid-drag the incremental feedback turns the jump into a visible teleport; window.screen()==None silently skips the move. Fractional point deltas are truncated (`d.dx as i32`, titlebar.rs:565-568) so slow Retina-trackpad drags stall. Drag-while-maximized moves the maximized frame instead of restore-then-drag. First ~5px swallowed by the gesture threshold.
**Fix sketch:** use performWindowDragWithEvent: from the titlebar node's mouseDown (removes all coordinate math), or convert via primary-screen maxY + pass NSEvent.mouseLocation as platform_screen_position; carry the fractional residual in the gesture manager.
**Verification:** manual-loop + multi-monitor coordinate bug CONFIRMED; CSD resize-edges-missing (related) CONFIRMED.

### (d) Gamepad on all windowing systems
**State:** DEAD in practice on every desktop backend despite a complete pipeline (gilrs → process-global channel → GamepadManager → GamepadInput event → CallbackInfo::get_gamepad_state, all real code).
**Root causes (all CONFIRMED):** (1) gilrs is polled ONLY inside regenerate_layout (common/layout.rs:891-892 is the sole call site) and every idle loop blocks (Windows WaitMessage run.rs:984-999, X11 select run.rs:1405-1435, macOS NSApp loop — its 16ms tickTimers only runs user timers, no gamepad poll) — a pad press cannot wake the loop, so button-press → callback never happens in an idle app. (2) Even when drained, the event only surfaces on the NEXT process_window_events pass triggered by unrelated OS input (drain runs after event determination; nothing schedules a follow-up pass). (3) iOS GCController / Android InputDevice backends are documented empty stubs (apple.rs:14-17, android.rs:14-16) while capability.rs:183-187 reports supported=true. Plus: no deadzone filtering, no rumble, no headless injection.
**Fix sketch:** dedicated poll thread (or ~16ms shell timer from ensure_started) pushing into the existing channel + waking the loop via the platform waker; request an event pass when drain reports changed; fix mobile capability report.
**Verification:** pump gap, next-pass latency, mobile stubs all CONFIRMED.

### (e) A11y incl. scroll-via-a11y actions and live regions
**State:** Broken differently per OS; only macOS is near-healthy.
**Root causes (all CONFIRMED):** (1) Windows: adapter drops accesskit QueuedEvents without calling raise() (windows/accessibility.rs:102-104) → NO UIA events ever fire — focus changes, property changes, AND live-region announcements; Narrator only sees data on fresh pull. (2) X11/Wayland: accesskit_unix's update_window_focus_state is never called anywhere (adapter starts is_window_focused:false forever, accesskit_unix-0.21.1 adapter.rs:122; set_focus stub at x11/accessibility.rs:115-118 has a false "automatically managed" comment) → Orca never receives focus events; live-region delivery possibly suppressed too. (3) Scroll-via-a11y is half-wired: the full inbound handler exists (layout/window.rs:4816-4874 — ScrollUp/Down/Left/Right/SetScrollOffset/ScrollToPoint/ScrollIntoView all implemented against scroll_manager) but the tree never advertises ANY scroll action or scroll_x/scroll_y (managers/a11y.rs only ever add_action Focus/Click/SetTextSelection/ReplaceSelectedText/SetValue) — screen readers have nothing to invoke. (4) Incremental text-edit updates (typing, caret moves) are computed (window.rs:5942) but never pushed to any OS adapter; screen readers hear edits only after the next full relayout. (5) Live regions: set_live(Polite) is wired (a11y.rs:516-518) and works on macOS; dead on Windows (needs the dropped raise()); PARTIAL on Linux (focus-state suppression risk); no aria-live attribute parsing anywhere. (6) AccessibilityInfo.supported_actions is a public api.json field that is never read (CONFIRMED). (7) Wayland adapter init itself is fine (shared LinuxAccessibilityAdapter, fixed at wayland/mod.rs:1234-1241).
**Fix sketch:** call .raise() on update_if_active's return (mirror macOS); call update_window_focus_state from X11 FocusIn/Out and Wayland keyboard enter/leave; advertise scroll actions + offsets on scrollable nodes in update_tree; push last_tree_update from each backend's frame tick gated on a11y_dirty; map supported_actions in build_node.
**Verification:** all five core claims CONFIRMED by the adversarial pass.
