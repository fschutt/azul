## The PLT-stub blocker (this session's last finding)

When `lift_function` was switched to `lift_with_transitive_deps` (M8.7c-2, uncommitted edit), the per-cb lift WORKS for the user's wrapper functions (`MyDataModelRefMut_create` etc.) but stops one level deeper:

```
[azul-web]   transitive[2]: lifting MyDataModelRefMut_create addr=0x..b6fc
[azul-web]     dep: sub_..bea0 → host=0x..bea0 resolved=cb_..bea0 [Noop] visited=false
```

`0x..bea0` is in the user binary's `__stubs` section — a PLT trampoline:

```
0000000100003ea0 (label "_AzRefCount_clone stub"):
    adrp x16, __got@PAGE
    ldr  x16, [x16, __got@PAGEOFF]
    br   x16
```

dladdr on a stub address returns the `cb_<hex>` fallback (no real symbol). The recursive lift correctly skips these (would lift garbage bytes otherwise), so the chain terminates one hop short of the real `AzRefCount_clone`.

## The user's resolving architecture (98%-context handoff message)

User-direction quote:

> so, in theory, we need to "compile" every single callback anyway
> on server startup, so that we're "ready" in case the user requests
> more C functions - after all we don't need to stream the callbacks
> to the user if the functions aren't needed. So, what we can do is
> to "pre-fill" the "resolve dependencies map" with the disassembled
> calls that we have to later assemble to wasm files anyway - we can
> get the "resolve order" from the api.json, i.e. first we'd resolve-
> and-compile all api.json functions (since we know that they have
> no internal dependencies) and then we compile-and-remill-transpile-
> prepare all user functions (since they can depend on user
> functionality). Then we transpile the rest (i.e. if the user code
> calls into some C function, edge case for now). Then we have (on
> server startup) every C function compiled as a separate wasm module
> and can later on request it on-the-fly, only streaming the things
> needed. And the one thing needed maybe is an "AzStyledDom_hydrate
> LayoutCache" so that we can init the headless window for WASM based
> hit-testing.

This bypasses the PLT-stub problem entirely: instead of CHASING dependencies discovered at lift time, **pre-compile every function at server startup** in the right order.

### Pre-compilation order (per user direction):

1. **All api.json functions first.** These are the libazul-resident framework functions (`AzRefAny_*`, `AzString_*`, `AzDom_*`, etc.). They live in libazul.dylib, their addresses come from `dlsym(name)`. They have NO internal dependencies within the user binary (only call other libazul functions). Lift each. Build a `name → lifted-wasm-bytes` map at server startup.

2. **All user functions next.** These come from walking the rendered StyledDom (callback fn-ptrs) + the layout cb itself + the AZ_REFLECT_* macro-generated wrappers (MyDataModel_*). Lift each. Their calls into libazul functions resolve via the pre-compiled map from step 1.

3. **User-to-user calls** (edge case for now). If user code calls another user function (not via AZ_REFLECT), recursively lift those too.

4. **Result**: a server-side map `{symbol_name: wasm_bytes_per_function}`. Each request for `/az/cb/<name>.wasm` serves the pre-compiled bytes. No more on-the-fly transitive lift; no more PLT-stub chase.

### The missing helper: `AzStyledDom_hydrateLayoutCache`

For wasm-side hit-test, the client needs a layout cache derived from the StyledDom. Server already computes this for the initial render; needs to be exposed as a function the wasm side can call. Likely a new `AzStyledDom_*` function in libazul that takes a StyledDom + viewport bounds + returns the cache.

## Lessons learned this session

1. **Modern LLVM treats attribute-less wasm imports as `memory(none)`** — opt DCEs the entire call chain. Fix: `memory(readwrite, inaccessiblemem: readwrite)` on every import declare. Doc'd inline + in `m8_4_lift_runtime_gap_2026_05_16.md`.
2. **wasm-ld `--import-memory` + `--import-table`** are needed for shared memory across mini + cb wasms. Without them each wasm has its own memory + writes are invisible across.
3. **`copy_from_slice` for variable-length copies compiles to `memcpy`** which lifts to a noop stub. Hand-roll byte-by-byte for code that has to lift cleanly.
4. **Constant-pool loads** (`adrp+ldr 0xad71f0`) are the worst lift case — addresses past wasm memory bounds. The pre-compile-all approach side-steps this because it puts CONSTANTS in the wasm data section directly.
5. **dladdr returns the closest exported symbol** — for PLT stubs or internal static fns, it returns `cb_<hex>` fallback. This kills naive transitive-lift. The pre-compile-all approach avoids it (we know the names from api.json).
6. **alwaysinline + linkonce_odr** for helper IR is fine, but the inliner can aggressively simplify the inlined copy past safety even with volatile stores.
7. **Heap-allocated state works**; static-storage globals via `adrp+add` don't (their native addresses don't translate to wasm-ld data-section offsets).
