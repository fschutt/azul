## The framing that matters

M8.8-1 retired 7 lift-pipeline hacks by introducing a
SymbolTable. M8.9-prep / M8.9 added the **infrastructure** for an
in-process lift pipeline — a C++ wrapper, static linking of
remill + LLVM + LLD into libazul.dylib (35 MB → 130 MB), and
Rust FFI bindings — gated behind `web-transpiler-static` feature
+ `AZ_NATIVE_REMILL=1` env var. **The in-process path is wired
but produces wrong IR.**

The fundamental finding from the last session: **macOS subprocess
spawn is ~1 ms, not 100 ms.** The 400 ms / fn cost in
`produce_object_for` is the actual LLVM work
(LoadArchSemantics + Optimize + llc + lld). In-process eliminates
the spawn but doesn't materially change per-fn cost on its own.
The real wins are:

  - **Batched lift**: share LoadArchSemantics (~30 ms) across N
    fns. 12 fns: ~50 ms total instead of 12 × 50 ms.
  - **Parallelization**: rayon par_iter on the dep set (~5×
    speedup on 8 cores).
  - **Self-contained binary**: ship libazul.dylib without external
    remill/opt/llc/wasm-ld at runtime.

This document covers all three, plus closing the in-process
divergence that gates them.

## The plan — 4 phases

### Phase 1 — Close the in-process lift divergence (blocker)

**The bug** (`AZ_NATIVE_REMILL=1` path):

  - 4-byte input (`ret`): byte-identical to subprocess. ✓
  - 48-byte input (AzStartup_alloc): `manager.traces.size() == 3`
    vs subprocess's `1`. The two extras are `sub_<bl_target_a>`
    and `sub_<bl_target_b>` — REAL function bodies, not just
    declares — for `bl` targets that are clearly OUT of our
    `LiftMemory` byte range (offsets +0x1687f8 and +0x1687dc from
    PC, far past the 48-byte input).

**Repro recipe**:

```bash
# Build with in-process enabled
cargo build -p azul-dll --release \
  --features "build-dll web web-transpiler web-transpiler-static" \
  --no-default-features

# Compile + run with in-process opted in
cd examples/c && cc -o hello-world.bin hello-world.c -lazul -L../../target/release -I../../dll
AZ_SKIP_LAYOUT_LIFT=1 AZ_NATIVE_REMILL=1 \
  DYLD_LIBRARY_PATH=../../target/release \
  AZ_BACKEND=web://127.0.0.1:8800 ./hello-world.bin &

# Inspect lift logs
grep "azul-mini\|wasm-ld" /tmp/azul-server.log  # → duplicate symbol errors
```

**AzStartup_alloc bytes** (for offline repro without the full server):

```
60010034 f44fbea9 fd7b01a9 fd430091
2a0003f3 9405a1fe aa1303e0 52800101
f7a10594 fd7b41a9 f44fc2a8 c0035fd6
```

  - +0x14: `0x9405a1fe` = `BL pc+0x1687f8` (out of range)
  - +0x20: `0x9405a1f7` = `BL pc+0x1687dc` (out of range)

Both `bl` targets fall outside the 48-byte input. My
`TryReadExecutableByte` correctly returns `false` for them. Yet
remill's TraceLifter still produces full lifted bodies for
`sub_10a79c954` + `sub_10a79c944` in `manager.traces`.

**Hypotheses ranked by likelihood**:

  1. **`TryReadExecutableByte` is called with `byte_out=nullptr` as
     a "trace-known" probe and remill's interpretation of `false`
     differs from what `SimpleTraceManager` in `Lift.cpp` does.**
     Lift.cpp's manager inherits a different default? Check
     remill/include/remill/BC/TraceLifter.h for the TraceManager
     virtual signatures + their semantics. A new `MemoryProvider`
     interface may have been added that we're missing.

  2. **`LoadArchSemantics` returns a module that ALREADY contains
     placeholder `sub_<X>` definitions** that get carried through.
     We dis-asm'd `aarch64.bc` and found 0 `sub_<X>` defines, but
     maybe `LoadArchSemantics` synthesizes some at load time
     (eagerly lifts well-known runtime addresses?).

  3. **`MoveFunctionIntoModule` recursively pulls in
     bl-target placeholder definitions** that remill's own logic
     creates as "placeholder traces" for unresolved bl targets.
     Lift.cpp filters via `if (lifted_entry.first ==
     FLAGS_entry_address)` for one purpose, but moves everything
     in `manager.traces` regardless. If `manager.traces` contains
     these placeholders, we move them all.

  4. **gflags not parsed** with the right flag-set. Last session
     added `google::ParseCommandLineFlags + google::InitGoogleLogging`
     to `initialize_llvm_targets()` but the divergence persisted.
     Maybe a SPECIFIC FLAGS_* needs setting (e.g.,
     `FLAGS_disable_register_alloc_hints` or similar) — check
     `remill::TraceLifter`'s source for any `FLAGS_<x>` reads.

**Debug recipe**:

```cpp
// Add to lift_inner after trace_lifter.Lift(address):
fprintf(stderr, "[az_remill] manager.traces:\n");
for (auto &kv : manager.traces) {
  auto *fn = kv.second;
  fprintf(stderr, "  0x%llx → %s (%s, %zu instructions)\n",
          (unsigned long long)kv.first,
          fn->getName().str().c_str(),
          fn->isDeclaration() ? "DECL" : "DEFN",
          fn->isDeclaration() ? 0 : fn->getInstructionCount());
}
```

Then run the standalone test against the 48-byte AzStartup_alloc
input. If extras have non-zero instruction counts, they were
genuinely lifted from somewhere — find the source. If they're
declarations, our move loop is the problem.

**Verification criterion**: for the 48-byte AzStartup_alloc
input, `manager.traces.size() == 1` AND the lifted IR contains
exactly ONE `define ptr @sub_<entry_hex>(`.

Also re-verify the 4-byte `ret` case to confirm no regression.

**Stop signs**: if the bug turns out to be in remill's published
behavior (e.g., they DO emit placeholder traces and Lift.cpp's
flow happens to elide them via some downstream filter), don't
patch remill — replicate Lift.cpp's filter in our wrapper. Read
`remill/bin/lift/Lift.cpp` carefully for any cleanup after the
move loop we might have missed.

lift trace divergence — <root cause>`.

### Phase 2 — Wire compile + wasm_link into the lift pipeline

Once Phase 1 lands, the in-process `lift` returns clean IR. Two
more subprocess calls remain in `produce_object_for` +
`link_objects_to_wasm`:

  - `opt -O2` + `llc -mtriple=wasm32`  →
    `native_remill::compile_to_wasm32_obj`
  - `wasm-ld --no-entry --allow-undefined ...`  →
    `native_remill::wasm_link`

The `llvm-link patched + helper → linked` step (the only
subprocess between lift and compile) can be handled two ways:

  - **(a)** Concatenate `patched_ir` + `helper_ir` text-side in
    Rust, strip duplicate `target datalayout` + `target triple`
    headers from `helper_ir`, and pass the combined IR string to
    `compile_to_wasm32_obj`. LLVM's `parseIR` accepts a single
    module per call; duplicate target headers WOULD make it
    reject. The header stripping is ~15 lines of Rust.

  - **(b)** Add a 4th C-ABI `az_remill_link_ir(ir_strs[]) →
    linked_ir` that uses `llvm::Linker::linkModules`. Cleaner but
    one more FFI fn.

Go with (a) for the smallest diff. (b) is the right long-term
shape once we add batched lift in Phase 3.

**Wire-up pattern** (mirrors the lift wire-up at
`transpiler_remill.rs:482` from commit 933200484):

```rust
let use_native = cfg!(feature = "web-transpiler-static")
    && std::env::var_os("AZ_NATIVE_REMILL").is_some();
if use_native {
    #[cfg(feature = "web-transpiler-static")]
    {
        let combined_ir = strip_helper_headers(&helper_ir).into_owned()
            + "\n" + &std::fs::read_to_string(&patched_ir_path)?;
        let obj_bytes = super::native_remill::compile_to_wasm32_obj(&combined_ir)
            .map_err(...)?;
        std::fs::write(&obj_path, &obj_bytes)?;
    }
} else {
    // existing llvm-link + opt + llc subprocess calls
}
```

Same shape for `link_objects_to_wasm`:

```rust
if use_native {
    let objs: Vec<Vec<u8>> = object_paths.iter()
        .map(|p| std::fs::read(p))
        .collect::<Result<_, _>>()?;
    let wasm_bytes = super::native_remill::wasm_link(
        &objs, exports,
        memory_mode == MemoryMode::ImportMemory,  // import_memory
        true,                                      // import_table (always)
        match memory_mode {
            MemoryMode::OwnMemory => 2 * 1024 * 1024,
            MemoryMode::ImportMemory => 2 * 1024 * 1024,
        },
    )?;
    std::fs::write(&wasm_path, &wasm_bytes)?;
}
```

**Verification**: e2e passes with BOTH paths:
- `AZ_NATIVE_REMILL=1` set: e2e green, lift logs show 0

- `AZ_NATIVE_REMILL=` unset: e2e green (regression check).

Layout-cb lift should drop from ~16 s to ~5-10 s on first
request (still slow because we're not batching yet — Phase 3).

### Phase 3 — Batched lift + rayon parallelization

Two compounding wins; do them together because the rayon work
depends on batched lift to amortize LoadArchSemantics correctly.

**Batched lift**: extend the C-ABI with a new function

```c
typedef struct {
    uint64_t address;
    const uint8_t *bytes;
    size_t bytes_len;
} AzLiftItem;

int az_remill_lift_batch(
    const char *arch_name,
    const char *os_name,
    const AzLiftItem *items,
    size_t items_len,
    char **ir_out,           // single combined IR string
    size_t *ir_len_out,
    char **err_out);
```

Implementation: ONE `LoadArchSemantics`, ONE `TraceManager` that
spans every item's byte range, loop over items calling
`trace_lifter.Lift(item.address)`, then ONE
`OptimizeModule` + ONE move-into-`dest_module` over every trace
collected. Output is one combined IR with N `define ptr
@sub_<canonical_addr_hex>` entries.

Per-fn cost should drop from ~50 ms to ~5 ms once
`LoadArchSemantics`' ~30-40 ms is shared across all items in the
batch.

**Rayon parallelization**: replace
`lift_with_transitive_deps`'s sequential while-loop with a
producer-consumer pattern. Pre-pass: scan each fn's bytes for
`bl imm26` instructions using the SymbolTable
(`bl_target = pc + sign_extend(imm26) * 4`), build the
unique-dep set with BFS, classify via
`table.lookup(addr).classification.is_recursable()`. Then
`par_iter` produces .o files. Cache + `FFI_LOCK` in
`native_remill` keep things correct.

```rust
// Pseudo:
let dep_set = pre_walk_deps_via_bytes_scan(table, roots);  // ~ms
let object_paths: Vec<PathBuf> = dep_set
    .par_iter()
    .map(|dep| produce_object_for_cached(dep))
    .collect::<Result<_>>()?;
let wasm = native_remill::wasm_link(...);
```

**Caveats for bytes-scan dep discovery**:

  - Direct `bl imm26` calls are caught.
  - Indirect calls (`blr X9`) are NOT caught — they go through
    `__az_call_indirect` machinery (helper IR provides body).
  - Rust trait-dispatch via vtables uses indirect calls — those
    don't end up as transitive deps anyway (they hit
    `__az_call_indirect` which classifies as `CallIndirect`).
  - Bare `b imm26` tail-call shims: caught by
    `detect_arm64_tail_shims` in the SymbolTable; chain map
    redirects to the real target, which IS in the dep set.

**Verification**:
  - Layout-cb lift: <2 s first request (40 fns × 50 ms parallel
    with LoadArchSemantics shared = ~5-10 ms each).
  - e2e: counter 5 → 12, no regression.
  - Lift logs: each unique canonical addr appears in
    `transitive[...]: lifting ...` exactly ONCE across the entire
    server lifetime.

- `web: M8.9-3b rayon parallel produce_object_for via dep-graph pre-walk`

### Phase 4 — Cross-platform support (Linux, then maybe Windows)

The macOS bits work. Linux needs:

  - `build.rs::build_in_process_remill` currently early-returns on
    non-apple targets. Mirror the macOS branch for Linux: the
    cxx-common bundle has a Linux variant; build it via
    `scripts/build_remill.sh` on a Linux host.
  - The C-ABI itself is portable (no Apple-specific bits in
    `azul_remill.cpp`).
  - lld driver: Linux uses `lld::elf::link` for non-wasm targets
    but our use case is wasm-only → `lld::wasm::link` is
    cross-platform, no change.
  - `symbol_table.rs` already has an `ingest_elf` path using
    `goblin::Object::Elf`. Verify `dl_iterate_phdr`-based image
    enumeration works (in-tree but untested).

Windows is M8.10 territory — defer unless explicitly requested.
