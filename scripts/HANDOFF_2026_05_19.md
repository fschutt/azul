## Architectural pieces in place

  1. **C-ABI pair pattern** for every function with a callback-wrapper
     arg. C user passes raw fn ptr; managed-FFI hosts call
     `*WithCtx` variant with `OptionRefAny` ctx. (M2.5)
  2. **Per-callback WASM serving** at `/az/cb/<sym>.<hash>.wasm`
     with `application/wasm` + `Cache-Control: immutable`. (M3 server)
  3. **Browser-side fetch + dispatch** via
     `WebAssembly.instantiateStreaming`, cached in `azWasmCache`
     Map. No server fallback POST. (M4 loader_js)
  4. **6-stage lift pipeline** in `RemillTranspiler::pipeline_single`:
     `remill-lift-17 → inject_alwaysinline → emit_helper_ir →
     llvm-link → opt -O2 → llc → wasm-ld`. (M5 + M6)
  5. **Per-callback-type wrapper synthesis** via `CallbackSignature`
     + `Pcs` enum + `signature_for_callback_kind`. Today: Callback
     shape covers every widget OnClick/Hover/etc.;
     CheckBoxOnToggle has its own (extra bool arg in X3); other
     kinds fall through with eprintln! warning. (M7-arch)
  6. **Symbol intercept pass** that dladdr-resolves every
     `sub_<hex>` branch destination + emits noop body stubs so
     produced WASM has 0 unresolved imports. (M7)

## What's genuinely working vs. mocked

**Genuinely working end-to-end:**
- C hello-world builds against the regenerated `azul.h` AS-WRITTEN
  (no source change needed beyond the M1 `AzDom_style` fix).
- Server starts, renders initial HTML, serves the per-callback
  WASM at content-hashed URLs.
- Lift pipeline turns real `.text` bytes into valid 462-byte WASM.
- Module instantiates in a real JS runtime (verified with Node;
  browser would behave identically).
- `callback(...)` is called from JS and returns a value.
- Dladdr correctly resolves the framework call destinations
  (logged at lift time).

**Mocked / no-op / TODO for M8+:**
- Framework calls (`MyDataModelRefMut_create`, `_downcastMut`,
  `_delete`) are stubbed as noop bodies → every framework call is
  effectively a no-op → the lifted callback's `if (!downcast)
  return DoNothing` early-exit fires regardless of input. Counter
  doesn't actually increment.
- `azul-mini.wasm` is still the 8-byte stub; no framework
  functions lifted into it.
- Server-side POST `/az/exec/` path works (M2.5 fixed the SIGBUS)
  but loader.js doesn't call it — by user direction.
- LayoutCallback (returns StyledDom = struct >16B) falls through
  to the Callback shape with eprintln! warning. Would need
  `Pcs::HiddenPtrReturn` for X8-hidden-return support.
- Panic/unwind: any panic call site in the lifted body becomes a
  `sub_<panic_addr>` extern → stubbed as noop → silently
  "succeeds" → control falls through. Real fix: import
  `__az_panic` from JS that throws. Documented in M6 commit.

## Findings worth preserving

Two memory notes capture concrete things future-you should know:

  1. `memory/m1_phase0_findings_2026_05_18.md` — the C/Rust ABI
     mismatch in `AzButton_setOnClick` (fixed by M2.5).
  2. `memory/m6_sroa_gap_2026_05_18.md` — why opt -O2 SROA does
     NOT evaporate the State alloca. Initial theory ("ptrtoint
     escape") was incomplete; the real reason is that the lifted
     body's `load i64 from state; inttoptr; load iN` pattern means
     state contents may alias arbitrary heap. SROA refuses to
     promote. Fix would need lift-side surgery (symbolic memory
     model or typed-pointer per-register-class) — ~1-2 days,
     deferred. Per-callback overhead ~462-982 bytes depending on
     body complexity, not the architectural-notes "~100 bytes".

### M8 — azul-mini.wasm as HeadlessWindow simulator (per user 2026-05-18)

The original M8 plan ("lift 200 framework functions individually
into azul-mini.wasm") was superseded by the user's "HeadlessWindow
simulator" direction:

  - WASM-side `azul-mini.wasm` runs the entire `azul-layout`
    event loop. Owns RefAny + parallel StyledDom arena.
  - JS-side `loader.js` is event source + DOM patch applier.
  - On click, JS dispatches `(x, y, button)` to a wasm export;
    wasm runs hit-test + EventFilter matching + dispatch (same
    logic as `desktop/shell2/common/dispatch_events_propagated`).
  - Per-callback WASM modules (M3-M7's output) become leaves the
    HeadlessWindow's dispatcher calls.
  - Framework calls inside callbacks (currently noop-stubbed)
    become typed imports from `azul-mini.wasm`'s exported surface.

Open architectural questions:
  - How to hydrate initial RefAny state in browser (script tag JSON
    vs separate fetch).
  - Patch-operation encoding for WASM → JS → DOM.
  - Shared linear memory model across modules
    (`--import-memory --shared` + a single `WebAssembly.Memory`).
  - The synthetic `azul_web_main` entry function to transpile via
    the same M5 pipeline (the user mentioned this as a candidate).
