# M8.8 — From hack-fixing to architectural cleanup

> **For the next Claude session.** Read this whole document first.
> Then read `scripts/HACKS_REVIEW_2026_05_16.md` for the root-cause
> analysis. The four stages below replace what was originally a
> flat 6-step fix list.

## The framing that matters

The previous session (M8.7c, commit `7530483aa`) shipped a working
hello-world counter that increments through a remill-lifted
on_click cb executing in WebAssembly. **The architectural risk is
retired**: native Rust → disasm → remill → llc → wasm → executes
correctly against shared memory across module boundaries, every
piece end-to-end. That's a category change from "is this possible"
to "is this production."

But getting there took 14 problems fixed in sequence (see
`scripts/HACKS_REVIEW_2026_05_16.md` § Root-cause analysis). On
review **11 of those 14 problems were the same architectural bug
surfacing 11 ways**:

> **The lift pipeline has no canonical source of truth for symbol
> identity.** Five subsystems each compute "what address is this
> / what name does it map to" independently. Reconciliation
> happens at wasm-ld link time, after `opt -O2` has already
> constant-folded across inconsistencies.

Every existing workaround is downstream of this gap:
`resolve_macos_arm64_stub`, `rewrite_tailcall_wrapper`, the `.N`
suffix handling, the PLT-stub thunk plumbing, the
`is_recursable_dep` name-prefix filter, the dropped helper-IR
noop bodies that JS Proxy then patches over, the synthetic
`lift_addr = 0x100000000 + i*0x1000` scheme for the eventloop
fns.

**Your job is to build the SymbolTable that eliminates the
underlying gap, not to fix 14 more downstream symptoms.** If you
catch yourself adding another workaround at the byte level or
the name-string level, you're below the abstraction the system
should be operating in — stop and ask whether the SymbolTable
needs to know one more thing.

## The plan — 4 stages, very different risk profiles

### Stage 1 — Build the SymbolTable (load-bearing refactor)

**Why this matters disproportionately.** Stage 1 is not "step 1
of 4." It's the step that determines whether the other three are
building on rock or sand. If the SymbolTable abstraction is wrong,
every later refactor risks dislodging it.

**The contract** (define this precisely in a new file
`dll/src/web/symbol_table.rs`):

```rust
pub struct SymbolTable {
    by_addr: BTreeMap<usize, SymbolEntry>,
    by_name: HashMap<String, usize>,
    /// Stub-addr → canonical (real) addr.
    /// For non-stub addresses, `chain[addr] == addr` (identity).
    chain: HashMap<usize, usize>,
}

pub struct SymbolEntry {
    /// dladdr name for code symbols, or synthetic
    /// `data_<hex>` for data, or `stub_<hex>` for unnamed stubs.
    pub canonical_name: String,
    /// Real entry-point address (post-PLT-chase).
    pub canonical_addr: usize,
    /// Exact size from (next_symbol_addr - this_addr). Not a guess.
    pub size: usize,
    /// Code bytes (slice of the loaded image's __TEXT).
    /// None for data symbols.
    pub bytes: Option<&'static [u8]>,
    pub kind: SymKind,
    pub classification: FnClass,
}

pub enum SymKind {
    Function,              // lift this
    Stub { target: usize }, // PLT trampoline; lift the target instead
    Data,                  // data symbol; for const-pool resolution (later)
}

pub enum FnClass {
    Recursable,             // lift + walk its bl targets
    BumpAlloc,              // __rust_alloc family — emit bump body
    CallIndirect,           // __az_call_indirect — emit call_indirect bridge
    ResolveCallback,        // __az_resolve_callback — emit JS-import bridge
    Leaf,                   // emit typed extern; let it become an env import
    NeverLift,              // panic if anyone reaches this (AzApp_run etc.)
}

impl SymbolTable {
    pub fn build_from_loaded_image() -> Result<Self, BuildError>;

    /// Returns the entry FOR THE GIVEN ADDRESS — without stub
    /// chasing. The entry's `kind` tells you if it's a stub.
    pub fn lookup(&self, addr: usize) -> Option<&SymbolEntry>;

    /// Follows the stub chain. For non-stub addrs, returns the
    /// entry at addr. For stubs, returns the entry at the
    /// chain target. Multi-hop (PLT → GOT → real) handled.
    pub fn resolve(&self, addr: usize) -> Option<&SymbolEntry>;

    /// What name should remill / wasm-ld use for this address?
    /// Always the canonical (post-chain) name.
    pub fn canonical_name_for(&self, addr: usize) -> Option<&str>;
}
```

**Build it with `goblin = "0.10"`** (already handles Mach-O / ELF
/ PE). Single backend, every platform falls out as a code path
within `goblin::Object::parse`.

On macOS:
- Find `_mh_execute_header` extern → mach_header_64.
- Walk LC_SYMTAB → all defined symbols (name + addr).
- Walk LC_DYSYMTAB.indirectsymoff + __TEXT.__stubs.reserved1 →
  stub_addr → indirect_idx → symbol_idx → name. Then
  `dlsym_self(name)` → real addr. Populate `chain`.
- Sort by addr; size = next_addr - this_addr.

On Linux:
- `dl_iterate_phdr` to walk loaded ELF segments.
- Parse .symtab + .dynsym + .got.plt the same way.

For non-loaded crates (libazul itself if dlopen'd separately),
walk via `dlopen` handle.

### Wire SymbolTable through the lift pipeline

Once the table exists, replace every ad-hoc lookup:

| current code | replacement |
|---|---|
| `resolve_fn_ptr(addr)` in `mod.rs` (dladdr + PLT chase) | `table.resolve(addr)` |
| `LIFT_READ_WINDOW` flat 4096 bytes | `table.lookup(addr).bytes.unwrap()` |
| `resolve_macos_arm64_stub` | delete; chain is precomputed |
| `rewrite_tailcall_wrapper` | delete; with exact `size` remill sees the full body; bare-`b` tail calls become a post-lift IR rewrite using the same thunk machinery |
| `branch_target_to_host_addr` + `.N` suffix handling | normalize remill's IR post-lift: rewrite every `sub_<hex>` in the lifted IR to `sub_<canonical_name>` based on table lookup. Dedup happens because two `bl X` to the same X now produce two calls to the same `sub_<canonical>` |
| `is_recursable_dep` name-pattern filter | `table.lookup(addr).classification == Recursable` |
| `lift_addr = native_addr` synthetic scheme for eventloop | use `entry.canonical_addr` consistently |
| Helper-IR dropped noop bodies + JS Proxy fallback | for each extern: look up classification; emit per-kind body OR typed extern declare. No more shape-guessed Proxy noops for sub_<hex> symbols |

### Stage 1 verification (don't skip)

The success criterion is **not** "the e2e demo still works" — that
might pass even if SymbolTable is internally wrong because the
existing workarounds mask it. The success criteria are:

1. Demo passes the existing e2e test (counter 5→12 over 7 clicks).
2. **`AzRefCount_canBeSharedMut` lifts without the byte
   rewriter.** Delete `rewrite_tailcall_wrapper`; rebuild;
   confirm canBeSharedMut still returns 1 (true) for a fresh
   refcount. If it doesn't, the SymbolTable's handling of bare
   tail-call shims is missing.
3. **`AzStartup_hydrate` lifts without the `.N` suffix
   handling.** Delete the `.N` handling from
   `parse_extern_sub_declares` + `branch_target_to_host_addr`;
   confirm hydrate's two `bl __rust_alloc` sites both bump the
   pointer. If they don't, the SymbolTable's symbol-name
   canonicalization didn't reach the IR rewrite step.
4. **No `cb_<hex>` placeholder names anywhere** in the lift logs.
   If any address resolves to `cb_<hex>`, the SymbolTable failed
   to cover it. Add it to the table or document why it's
   intentionally unknown.
5. **No env imports of the form `env.sub_<hex>`** in the
   generated wasm modules. The Proxy still serves the real
   `env.memory` / `env.__indirect_function_table` /
   `env.__az_resolve_callback` etc., but `sub_<hex>` imports
   indicate a missed symbol classification.

If criteria 2-5 fail individually, **don't paper over them with
a new workaround**. Add what's missing to the SymbolTable. The
whole point of Stage 1 is to retire the workaround pattern.

**Stage 1 commit message convention**:
`web: M8.8-1 SymbolTable — replaces N byte/name hacks`.

### Stage 2 — Layout-cb-executes probe (cheap insurance, ~50 LOC)

**Why this before Stage 3.** The lifted layout callback has never
been instantiated. Layout cbs are categorically heavier than
on_click: they allocate `Vec<NodeData>`, construct `AzString`s
via `AzString_copyFromBytes` (heap alloc), call into
`AzButton_create` / `AzDom_addChild`, store fn-ptrs in struct
fields (`AzButton.on_click: Callback`). At least four classes of
failure are possible, each potentially M6/M8.4b-class:

1. **Bump allocator can't satisfy Vec resizes** (Vec grows by
   alloc+copy+free; bump can't free → fragmentation after a few
   resizes).
2. **call_indirect roundtrip on stored fn-ptr** (store ptr in
   struct field, read it back, call through table). Theoretically
   wired but not exercised.
3. **Multiple cb-wasm modules' static-data sections collide** in
   the shared memory. on_click cb doesn't touch much static
   data; layout cb might.
4. **Lift fidelity issues on the Box<TaggedEnum> pattern** azul's
   `Dom` uses extensively.

A 50-line probe that just invokes the layout cb's exported
`callback()` once and observes what happens tells you which of
these (if any) is broken **before** you build hit-test + diff/patch
on top of it. Building Stage 3 on a broken Stage 2 foundation is
the M8.7c trap repeated at a larger scale.

### The probe

Create `/tmp/layout-probe.js`:

```javascript
const http = require("http");

async function fetch_(p) {
  return new Promise((res, rej) =>
    http.get("http://127.0.0.1:8800" + p, r => {
      const c = []; r.on("data", x => c.push(x));
      r.on("end", () => res(Buffer.concat(c))); r.on("error", rej);
    })
  );
}

(async () => {
  const html = (await fetch_("/")).toString();
  const typeId = BigInt(html.match(/"type_id":"(\d+)"/)[1]);
  const counter = parseInt(html.match(/<div id="az_1">(\d+)<\/div>/)[1]);
  const miniUrl = html.match(/href="(\/az\/mini\.[^"]+)"/)[1];
  const layoutUrl = html.match(/href="(\/az\/layout\/[^"]+)"/)[1];

  const miniBytes = await fetch_(miniUrl);
  const layoutBytes = await fetch_(layoutUrl);

  const table = new WebAssembly.Table({ initial: 64, element: "anyfunc" });
  const realEnv = { __indirect_function_table: table, __az_resolve_callback: () => 0xFFFFFFFF };
  const stubFor = n => /write_memory|barrier|exception_clear/.test(n)
    ? (() => {}) : (/(?:_64|_f64)\b/.test(n) ? (() => 0n) : (() => 0));
  const h = env => ({ get: (_, p) => typeof p === "string"
    ? (Object.prototype.hasOwnProperty.call(env, p) ? env[p] : stubFor(p)) : undefined,
    has: () => true });

  const { instance: miniI } = await WebAssembly.instantiate(miniBytes, { env: new Proxy({}, h(realEnv)) });
  const mini = miniI.exports;
  const memory = mini.memory;
  mini.AzStartup_init(0, 0);
  const modelPtr = mini.AzStartup_alloc(4);
  new DataView(memory.buffer).setUint32(modelPtr, counter, true);
  const refanyPtr = mini.AzStartup_hydrate(
    Number(typeId & 0xFFFFFFFFn),
    Number((typeId >> 32n) & 0xFFFFFFFFn),
    modelPtr, 4);
  console.log("refany=" + refanyPtr);

  const layoutEnv = { memory, __indirect_function_table: table };
  let layoutI;
  try {
    layoutI = (await WebAssembly.instantiate(layoutBytes, { env: new Proxy({}, h(layoutEnv)) })).instance;
  } catch (e) {
    console.log("LAYOUT INSTANTIATE FAILED:", e.message);
    process.exit(1);
  }
  console.log("layout instantiated; exports:", Object.keys(layoutI.exports).slice(0, 5));

  // Layout cb signature: returns AzStyledDom (>16B) — uses hidden X8.
  // Current wrapper sig doesn't model X8. Best we can do for the
  // probe: pass an out-buffer as info_ptr and observe whether the
  // body executes at all. Trapping tells us where; "returns
  // something nonzero" without trapping is the green light.
  const outBuf = mini.AzStartup_alloc(4096);
  const dv = new DataView(memory.buffer);
  const bumpBefore = dv.getUint32(modelPtr + 4, true);  // just record some other slot

  let result, trapMsg = null;
  try {
    result = layoutI.exports.callback(BigInt(refanyPtr), 0n, outBuf);
  } catch (e) {
    trapMsg = e.message;
  }

  if (trapMsg) {
    console.log("TRAPPED:", trapMsg);
    // Inspect bump pointer + outBuf bytes to learn how deep we got.
    process.exit(2);
  }

  console.log("layout returned:", result, "without trapping");
  // Sample outBuf — if the lifted body actually wrote a StyledDom
  // shape we'd see non-zero bytes near the start.
  const sample = [];
  for (let i = 0; i < 64; i += 8) sample.push(dv.getBigUint64(outBuf + i, true).toString(16));
  console.log("outBuf[0..64] as u64:", sample.join(" "));
  process.exit(0);
})().catch(e => { console.error(e.stack); process.exit(1); });
```

### Probe outcomes (decision matrix)

| outcome | meaning | next move |
|---|---|---|
| returns nonzero ptr, outBuf has plausible StyledDom shape | layout cb lift works end-to-end | proceed to Stage 3 as planned |
| returns 0, no trap | lift produces valid wasm but body short-circuited | inspect lifted IR; usually a missing recursable classification; fix in SymbolTable Stage 1 not as a new hack |
| traps with "memory access out of bounds" early | pointer arithmetic landed outside the 2 MiB initial memory | grow memory or hoist const-pool data |
| traps with "stack overflow" | 4 KiB wrapper stack alloca insufficient for layout cb depth | size the alloca from SP-displacement analysis (hack #13) |
| hangs | bump allocator fragmentation after Vec resizes; or infinite recursion | replace bump with a real wasm-targeted allocator (dlmalloc/wee_alloc); investigation needed |
| traps with "call_indirect signature mismatch" | the fn-ptr roundtrip went through but the table slot has the wrong sig | wrapper signature generalization (Stage 4 hack #9) needs to come forward |
| returns garbage | lift fidelity on Vec/Box; deep investigation needed | this is the M6-class outcome — page the user; do NOT silently rewrite layout cb to dodge it |

**Stage 2 commit message convention**:
`web: M8.8-2 layout-cb probe — <outcome>`.

### Stage 3 — Second subsystem: dispatch + hit-test + diff/patch

This is the genuinely new work. It looks like "fixing hacks" but
it's actually building a subsystem that has never run. Treat the
risk accordingly.

Six catalog items land together because they're interdependent:

1. **`AzStartup_setAppData(state, refany_ptr)`** in
   `eventloop.rs`. JS calls it after `AzStartup_hydrate` so
   mini's eventloop has the hydrated refany ptr instead of
   `FAKE_REFANY_LO = 0x101`.

2. **`AzStartup_dispatchEvent`** uses
   `state.app_data_ptr` from above. The cb invocation via
   `__az_call_indirect` passes the real refany (lo + hi).

3. **`cb_fn_cache`** in mini state — `HashMap<node_idx,
   fn_addr>`. Populated from the hydrated StyledDom (Step 4
   below adds the StyledDom hydration; until then, populate
   from the same `data-az-cb` walk JS already does).

4. **WASM-side hit-test** in `AzStartup_dispatchEvent`. Walks
   the hydrated StyledDom's per-node bounding boxes for (x,y),
   returns node_idx. Replaces JS-side `azNodeIdxFromEvent`.

5. **`AzStyledDom_serialize` / `_deserialize`** (or postcard).
   Server serializes the StyledDom; embeds as base64 in
   `<script id="az-styled-dom">`. Wasm-side `AzStartup_init`
   deserializes into `state.current_dom`.

6. **Diff + patch loop**. After cb returns `Update::RefreshDom`,
   mini invokes the lifted layout cb (the one Stage 2 probed),
   diffs against `state.current_dom`, emits TLV patch stream into
   a buffer, returns the buffer ptr + length. JS reads the buffer
   via the existing `azApplyPatches` decoder.

7. **Restore `azWireListeners` to bind all event kinds** (click,
   mousedown, mouseup, mousemove, dblclick, wheel, keydown,
   keyup, focusin, focusout, resize, scroll). Each calls
   `mini.AzStartup_dispatchEvent` with the kind + event buffer.

8. **Delete `azInvokeCbDirect`** and the hardcoded
   `getElementById('az_1').textContent = ...`. The flow becomes
   wasm-dispatch all the way through.

9. **Bundled CSS shipping**. Whatever default theme the desktop
   renderer uses (azul-css's native button style etc.) gets
   embedded into the rendered HTML's `<style>` block so the
   button looks the same on first paint.

**Stage 3 verification**: a click in a real browser increments
the counter visibly WITHOUT any hardcoded element ID in
loader.js, AND a second clickable element (add a "reset" button
to hello-world.c) fires the right cb when clicked.

- `web: M8.8-3d StyledDom serialize/deserialize`
- `web: M8.8-3e diff + patch emission`
- `web: M8.8-3f restored full event wiring`

### Stage 4 — Mechanical cleanup

After Stages 1-3 the remaining items are small and largely
independent:

- **Lifted `_fromJson` hydration** (hacks #11, #12, #18). Replace
  `AzStartup_hydrate`'s hand-rolled RefAny tree with an
  invocation of the user's lifted `_fromJson` via
  `__az_call_indirect`. The wrapper synthesis from Stage 4 next
  bullet is the prerequisite.
- **api.json-driven wrapper signatures** (hack #9). Replace
  `signature_for_callback_kind`'s hand-rolled per-kind sigs with
  a generic `synthesize_wrapper_for_api_fn` driven by api.json
  type info. Adds `Pcs::HiddenPtrReturn` for X8 aggregate returns
  (which the layout cb needs anyway).
- **SP-displacement analysis** (hack #13). Scan the lifted IR
  for max stack use; size the wrapper alloca accordingly.
- **Const-pool data hoisting** (the residual M8.7c hack #6 — only
  if needed). If anything beyond hello-world hits the Box::new
  const-pool issue, extend SymbolTable to cover data sections
  and rewrite `adrp+ldr` against data symbols to wasm data-section
  loads.

`web: M8.8-4 <item>`.

## What NOT to break

The following are intentional and correct; SymbolTable should
preserve their shape, not replace them:

- **The api.json walk** in `classify.rs` (drives off the brotli-
  compressed embedded JSON, no hand-coded fn list). SymbolTable
  *consumes* this classification — it doesn't replace it.
- **The bump allocator** in `transpiler_remill.rs::emit_helper_ir`
  (`@__az_bump_ptr` global, RustAlloc kind). Stage 2 may tell
  you it's insufficient, in which case replace it; until then,
  leave it.
- **`--import-memory` + `--import-table`** for cb / layout
  wasms. Shared memory architecture, correct.
- **PLT-stub THUNK emission** (current `sub_<stub> →
  musttail sub_<real>` thunks via
  `ResolvedBranchExtern.real_addr`). Stage 1 populates `real_addr`
  from the SymbolTable's `chain` map instead of from the byte
  parse, but the thunk pattern itself is the right shape and
  generalizes to tail-call shims too.
- **Pre-validation of RefAny serializer at startup** in
  `headless.rs::HeadlessApp::validate`. Fail-fast for
  misconfigured apps.
- **Server-side fallback** at `POST /az/exec/<node_id>`. Leave
  it; Stage 3 will add a `?fallback=server` URL param to take
  this path on wasm-unsupported browsers.
