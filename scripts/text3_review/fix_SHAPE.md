# Fix brief — cluster SHAPE  (10 confirmed bugs)

Each bug was adversarially verified. Fix the ROOT CAUSE per the sketch; keep the
public API stable unless the fix requires it (note any signature change). After all
fixes, the crate MUST compile.

## 1. [HIGH] `layout/src/text3/default.rs:774`
**Byte offset stored in RawGlyph.liga_component_pos breaks allsorts mark-to-mark (mkmk) and mark-to-ligature positioning**

- **What:** shape_text_internal repurposes allsorts' `liga_component_pos` field to carry the per-glyph BYTE offset (`liga_component_pos: cluster as u16`, and read back at default.rs:858). But allsorts' GPOS uses this field as a ligature-COMPONENT index. In `forall_mark_mark_glyph_pairs` (third_party/allsorts/src/gpos.rs:714) mark-to-mark attachment is gated on `infos[i].liga_component_pos == infos[j].liga_component_pos`. Upstream, all base/mark glyphs default to 0 so the test passes; here every glyph carries its distinct byte offset, so two stacked combining marks have unequal values and the mkmk lookup is silently skipped (the ligature() fallback on line 716 also doesn't fire for ordinary marks). markligpos (gpos.rs:818) likewise receives the byte offset instead of the component index, selecting the wrong ligature-component anchor.
- **Repro:** Shape a 2+ combining-mark cluster with a font that has an mkmk GPOS lookup, e.g. Vietnamese "ệ" (U+0065 e + U+0302 combining circumflex + U+0323 combining dot below) or Arabic "لّ" (base + shadda + another haraka). raw_glyphs get liga_component_pos = byte offsets 0,1,3 (approx). init_from_glyphs marks both diacritics is_mark=true. In forall_mark_mark_glyph_pairs, i=circumflex(pos=1), j=dot-below(pos=3): 1 != 3 and neither .ligature() → markmarkpos NOT applied. Result: the second mark keeps only its markbasepos placement (or none) and overlaps the first instead of stacking, diverging from CoreText. Fix: carry the byte cluster in RawGlyph.extra_data (the () type param via a GlyphData impl) or a parallel side table, leaving liga_component_pos at allsorts' managed default so the GPOS equality holds.
- **Fix:** Do not overload liga_component_pos. Track byte offsets in a side table (parallel Vec keyed by glyph identity) or use RawGlyph.extra_data (the `()` type param) to carry the byte cluster, leaving liga_component_pos at its allsorts-managed default so GPOS mkmk/marklig equality works.

## 2. [MEDIUM] `layout/src/text3/default.rs:770`
**Glyphs at byte offset > 65535 are silently dropped from shaping**

- **What:** The RawGlyph build loop only pushes a glyph when `u16::try_from(cluster).is_ok()` because the byte offset is stuffed into the u16 liga_component_pos. Any character whose byte offset exceeds 65535 is skipped entirely.
- **Repro:** A single large text run / coalesced merged_text over 64KB (large <pre> block or a long paragraph, and note shape_visual_items merges adjacent same-style runs into one merged_text) loses every glyph past byte 65535 — the tail of the text renders as nothing.
- **Fix:** Store the byte cluster in a wider side-channel (see finding #1) rather than a u16, or at minimum shape in <64KB chunks with an offset base.

## 3. [MEDIUM] `layout/src/text3/default.rs:790`
**Setting any font-feature or font-variant disables default ligatures/contextual-alternates for Latin-family scripts**

- **What:** When `user_features` is non-empty the shaper switches to `(FeatureMask::empty(), user_features)` (default.rs:789-794), discarding the default mask (LIGA, CLIG, CALT, CCMP, LOCL, RLIG). For ScriptType::Default the vendored gsub only re-adds CCMP|RLIG|LOCL (third_party/allsorts/src/gsub.rs:1495), so LIGA/CLIG/CALT stay off. add_variant_features also pushes into user_features, so any font-variant triggers the same path.
- **Repro:** Shape "office fluff" with a font containing fi/ff/ffl ligatures (e.g. a typical Latin text font). Call once with default StyleProperties, and once with style.font_variant_numeric = OldstyleNums (or style.font_features = ["ss01"], or font_variant_caps = SmallCaps). Compare output Glyph vectors: the default run substitutes fi/ff/ffl ligature glyphs (fewer glyphs, different advances); the variant run keeps separate f/i/f/l glyphs because user_features is non-empty -> feature_mask=empty -> LIGA/CLIG/CALT never applied. Total run width and glyph ids differ.
- **Fix:** Start from build_feature_mask_for_script's default mask and additively merge user features/alternates (subtracting only features the user explicitly set to 0) instead of replacing the mask with empty.

## 4. [MEDIUM] `layout/src/text3/default.rs:866`
**Hinted advance uses rounded integer ppem while glyph offsets/kerning use the exact (fractional) font size**

- **What:** `ppem = font_size.round() as u16` is passed to get_hinted_advance_px, whose returned advance is scaled by the rounded ppem and grid-rounded to a whole pixel. Meanwhile GPOS offsets and kerning are scaled by `scale_factor = font_size / units_per_em` using the UNROUNDED font_size (default.rs:838, 870, 878). When hinting is unavailable the fallback advance also uses the unrounded scale_factor (line 869). Advances and positions therefore correspond to two different font sizes.
- **Repro:** Shape a run at font_size_px = 14.4 (e.g. 0.9em of 16px under DPI scaling) with a font whose upem=1000 and hint_instance present. Take a glyph with horz_advance=500 funits. Fallback/positioning basis: scale_factor=14.4/1000=0.0144 → fractional advance would be 7.2px, and kerning/GPOS offsets are computed at this 14.4 basis. Hinted path: ppem=round(14.4)=14, scaled advance = 500*14/1000 = 7.0px, grid-rounded to whole pixel → 7.0px. Result: advance=7.0 (14px basis) but kerning/mark offsets at 14.4 basis → per-glyph ~0.2px (2.8%) narrower advance plus basis-mismatched mark placement; disable hinting (or use a glyph with no outline) and the same glyph now advances 7.2px, showing the hinted/fallback discontinuity.
- **Fix:** Either scale the hinted advance back to the exact font_size (advance_px * font_size / ppem_rounded) or use unrounded scaling consistently for advance, kerning and offsets; at minimum keep the advance basis identical between the hinted and fallback branches.

## 5. [MEDIUM] `layout/src/text3/edit.rs:120`
**Multi-cursor edits that add/remove runs leave other cursors' source_run indices stale**

- **What:** edit_text processes selections back-to-front and calls adjust_cursors to shift already-processed cursors, but adjust_cursors only shifts start_byte_in_run for cursors whose source_run == edit_run. When an earlier-in-document edit changes the NUMBER of runs (delete_range merging two runs into one, or a cross-run backspace/forward-delete removing a run), the source_run of every later run shifts by one, but the already-processed (later-in-doc) cursors are never reindexed. Their source_run now points at the wrong run.
- **Repro:** content = [r0, r1, r2, r3, r4], all InlineContent::Text with identical style (so they merge). Two selections: cursor A = Range spanning run 2..run 3 (multi-run delete), cursor B = Cursor in run 4 with an Insert edit. Call edit_text(content, [A, B], edit). Back-to-front: B (run 4) is processed first, inserts into run index 4, and is pushed to new_selections with source_run=4. Then A is processed: delete_range removes/merges the run at index 3, so new_content shrinks to 4 runs (indices 0..3) and the former run-4 content is now at index 3. adjust_cursors(edit_run=2, ...) leaves B untouched (4 != 2). Result: B still has source_run=4, now out of bounds (only 0..3 exist) — the second caret lands on the wrong run or is silently dropped on the next edit/render.
- **Fix:** After an edit that changes new_content.len(), decrement source_run of all previously-processed cursors whose source_run > the removed/merged run index (track the delta in run count, not just byte length).

## 6. [MEDIUM] `layout/src/text3/edit.rs:224`
**Deleting or type-replacing a selected non-text item (image) is silently a no-op**

- **What:** In delete_range, the single-run path only mutates when new_content.get_mut(start_run_idx) is InlineContent::Text; if the selected run is a non-text item (image/object) nothing is removed and the caret collapse code never runs. delete_backward/delete_forward likewise only act on Text runs and won't merge across a non-text neighbor. So a selection that is exactly an inline image, or backspacing/forward-deleting into an image, does nothing.
- **Repro:** content = [Text("a"), Image(img), Text("b")] (indices 0,1,2). Build a Range selection over the image: start = TextCursor{cluster_id:{source_run:1,start_byte_in_run:0}, affinity:Leading}, end = same cluster_id with affinity:Trailing (exactly what hittest_cursor yields dragging across the image). Call apply_edit_to_selection(content, &Selection::Range(range), &TextEdit::Insert("x")) or &TextEdit::DeleteForward. delete_range takes start_run_idx==end_run_idx==1, get_mut(1) is InlineContent::Image so the `if let Some(Text)` body is skipped; result content is unchanged and the image is still present — Delete does nothing and the typed 'x' is lost. Equivalent: caret Trailing on the image (source_run:1) + DeleteBackward, or caret Leading on Text("b") (source_run:2,byte:0) + DeleteBackward, both return content unchanged.
- **Fix:** Handle non-text runs in delete_range (remove the whole item when the selection covers it) and in delete_backward/forward (remove or step over an adjacent non-text run) instead of only matching InlineContent::Text.

## 7. [MEDIUM] `layout/src/text3/glyphs.rs:147`
**CombinedBlock (tate-chu-yoko) glyphs all render stacked at one x in the primary render path**

- **What:** In get_glyph_runs_simple, the CombinedBlock arm calls the pen-advancing `process_glyphs` closure once PER glyph, each time passing `item.position.x` as `item_origin_x`. The closure resets `pen_x = item_origin_x` at the top of every call, so each glyph is emitted at `item.position.x + glyph.offset.x` with the pen advance (`pen_x += glyph.advance + glyph.kerning`) thrown away between calls. The glyphs of a combined block therefore never advance relative to each other. The correct sibling function `get_glyph_positions` (lines 461-464) passes the WHOLE `glyphs` slice in one call so the pen advances. CombinedBlock glyphs carry raw GPOS offsets (offset.x ~= 0 for Latin digits, see construction at cache.rs:7348-7364), not pre-accumulated pen positions, so nothing compensates.
- **Repro:** Vertical Japanese text (writing-mode: vertical-rl) containing an element with `text-combine-upright: all` wrapping a 2+ char run such as "24" or "2024". Shaping produces a ShapedItem::CombinedBlock with N glyphs each having its own advance and offset.x≈0. Run through solver3 display_list get_glyph_runs_simple. Inspect the emitted SimpleGlyphRun glyph instances: all N glyphs have absolute_position.x == item.position.x (+ tiny GPOS offset), i.e. identical x, instead of x increasing by cumulative advance. Compare with get_glyph_positions on the same layout, which yields correctly spread x coordinates — the divergence between the two functions is the bug.
- **Fix:** Mirror get_glyph_positions: call process_glyphs(glyphs, item.position.x, glyphs.first().map_or(default, |g| g.style.writing_mode), None) ONCE on the full slice instead of looping per glyph.

## 8. [MEDIUM] `layout/src/text3/script.rs:489`
**is_hangul swallows Halfwidth/Fullwidth Forms and Enclosed CJK, misclassifying kana/fullwidth Latin as Hangul**

- **What:** is_hangul matches `'\u{3200}'..='\u{32FF}'` (Enclosed CJK Letters and Months) and `'\u{FF00}'..='\u{FFEF}'` (Halfwidth and Fullwidth Forms). Those blocks contain fullwidth ASCII/Latin, fullwidth digits, fullwidth CJK punctuation (！？，。（）), and the Halfwidth Katakana sub-block U+FF61-FF9F — none of which are Hangul. Because Hangul is checked before Katakana/Mandarin in SCRIPT_CHECKERS (line 150 vs 152/143), these characters can never reach the Katakana or Mandarin checker.
- **Repro:** detect_script("\u{FF71}\u{FF72}\u{FF73}") (halfwidth katakana "ｱｲｳ") returns Some(Script::Hangul); likewise detect_script("\u{FF21}\u{FF22}\u{FF23}") (fullwidth "ＡＢＣ") returns Some(Script::Hangul). Each char is a non-stop char that first matches is_hangul (FF00-FFEF), increments the Hangul counter, and Hangul is checked before Katakana/Latin so no other checker fires. The Script then drives shaping/font fallback via VisualItem in cache.rs:6518/6532.
- **Fix:** Remove U+3200-32FF and U+FF00-FFEF from is_hangul; route U+FF61-FF9F to is_katakana and leave fullwidth Latin/punct to the appropriate checkers (or treat fullwidth ASCII as stop/Latin).

## 9. [MEDIUM] `layout/src/text3/selection.rs:121`
**Double-click word selection concatenates clusters in visual order and only within one line_index, breaking on bidi and soft wraps**

- **What:** select_word_at_cursor builds line_text by pushing cluster text in layout.items order (visual order) for the cursor's line_index, computes the cursor's byte offset by summing cluster lengths until the matching cluster_id, then runs find_word_boundaries on that concatenation. For bidi text the visual order differs from logical order, so word boundaries are computed on scrambled text and map back to the wrong clusters. Because it filters by a single line_index, a word split across a soft wrap is only partially selected.
- **Repro:** Soft-wrap (deterministic): lay out an editable text run containing a single long word (e.g. "supercalifragilisticexpialidocious") in a container narrow enough to force overflow-wrap: break-word to split it across two lines. The word's clusters now span line_index N and N+1. Double-click a cluster on line N and call select_word_at_cursor: extract_line_text_and_clusters gathers only line-N clusters, find_word_boundaries returns word_end == fragment length, and the returned SelectionRange covers only the line-N fragment (both endpoints on line N). Expected: whole word across both lines highlighted. Bidi (illustrative): LTR paragraph with a space/period-delimited token where an embedded number or nested RTL segment reorders the boundary char between logical and visual order; the maximal visual word-char run then maps to a cluster set that differs from the logical word.
- **Fix:** Concatenate clusters in logical order (sort by source_cluster_id / logical byte) spanning the whole run, not just one visual line, before running word segmentation; map results back to cluster ids in logical space.

## 10. [LOW] `layout/src/text3/default.rs:890`
**Ligature glyph reports logical_byte_len of only its first component character**

- **What:** For a ligature, the surviving base glyph keeps the first component's byte offset, and logical_byte_len is set to `source_char.len_utf8()` — the length of just the first character. The remaining component bytes are unaccounted for in the glyph's logical span.
- **Repro:** Shape the two-char string "fi" with a font whose GSUB has the standard fi ligature under the default liga feature (e.g. DejaVu Sans / most text fonts). Call shape_text_internal and inspect the returned Vec<Glyph>: it contains a single ligature glyph with logical_byte_index=0 and logical_byte_len=1, whereas the ligature spans bytes 0..2 (expected logical_byte_len=2, matching the test model's lig_len). Assert glyph[0].logical_byte_len == 2 to see it fail.
- **Fix:** Compute logical_byte_len from the span covered by all merged unicodes (info.glyph.unicodes) or the distance to the next cluster's byte offset, not just the first char.
