# Fix brief — cluster TEXTLAYOUT  (18 confirmed bugs)

Each bug was adversarially verified. Fix the ROOT CAUSE per the sketch; keep the

## 1. [CRITICAL] `layout/src/solver3/getters.rs:2690`
**Pixel line-height (`line-height: 24px`) becomes a NEGATIVE per-run line height via the compact-cache fast path**

- **What:** CSS `line-height: <length>` is stored by the parser as a NEGATIVE PercentageValue (convention: negative normalized() = absolute px, see css/src/props/style/text.rs:1527-1536). The compact-cache encoder faithfully stores that negative value (core/src/compact.rs:1055 and :387 do `pct_x10 = normalized()*1000`, so `24px` -> normalized -24 -> stored -24000). But the fast path in get_style_properties does NOT decode the negative convention: it blindly computes `LineHeight::Px(normalized / 100.0 * font_size)`. For `line-height: 24px` with font_size 16 this yields `LineHeight::Px(-2400/100*16) = Px(-384)`. The slow path and fc.rs::translate_to_text3_constraints (fc.rs:3966-3969) both handle the `if n < 0 { -n }` convention correctly; only this per-run fast path is broken. Any px line-height <= ~32px (larger ones get clamped to sentinel = 'normal' at compact.rs:1058) produces a negative Px line height that LineHeight::resolve returns verbatim (cache.rs:131), so per-run line boxes get a negative/zero height and collapse or overlap.
- **Repro:** Render `<p style="line-height: 24px">Hello</p>` in the normal (non-hover) state with a compact cache populated (the default fast path). font-size resolves to 16px. get_style_properties -> line_height fast path returns LineHeight::Px(-384). At cache.rs:2113 the run's line-box height resolves to -384px, so lines overlap/collapse. Compare against the fc.rs container path which correctly yields Px(24). Any px line-height in (0, ~32.7]px triggers the negative result; > ~32.7px hits the i16 sentinel and is wrongly treated as `normal`.
- **Fix:** In the fast path, apply the same negative-is-absolute-px convention: `let raw = normalized; if raw < 0.0 { LineHeight::Px(-raw/10.0 /*decoded px*/) } else { LineHeight::Px(raw/100.0*font_size) }`. Note the decode already divided by 10; reconcile so a stored -24000 -> decoded -2400 -> +24px. Better: store an explicit is_px flag in the compact cache instead of overloading sign.

## 2. [HIGH] `layout/src/text3/cache.rs:4655`
**move_cursor_left/right ignore current affinity → one keypress swallowed; document start unreachable by Left arrow**

- **What:** move_cursor_right always returns the NEXT cluster with Leading affinity and move_cursor_left always returns the PREVIOUS cluster with Trailing affinity, ignoring the current cursor's affinity. A caret is a boundary between clusters, and Leading(N) and Trailing(N-1) denote the SAME visual boundary (get_cursor_rect maps Trailing(N-1) to N-1.x+advance and Leading(N) to N.position.x, which are equal in contiguous LTR text). So moving from a boundary whose representation already points in the travel direction produces no visual movement for that keypress. Additionally, from Trailing(first cluster) a Left press finds no previous cluster and returns the cursor unchanged, so Leading(first cluster) (the document/line start) can never be reached with the Left arrow.
- **Repro:** Layout text \"abc\" (3 LTR clusters, advances ~10px, positions a.x=0,b.x=10,c.x=20). (1) Swallowed keypress: click the right half of 'a' -> hittest_cursor yields TextCursor{cluster a, Trailing} (x=10). Press Right -> move_cursor_right returns {cluster b, Leading} (x=10). get_cursor_rect x is unchanged (10==10): caret does not move. Second Right press -> {cluster c, Leading} x=20, now it moves. (2) Start unreachable: with caret at {cluster a, Trailing} (x=10), press Left repeatedly -> move_cursor_left finds no previous cluster (current_pos=0), returns cursor unchanged; Leading(a) at x=0 (document start) can never be reached by Left.
- **Fix:** Branch on current affinity: for Right, if affinity==Leading advance to next cluster Leading, if Trailing the current position is already a boundary so advance two boundaries (next cluster's Trailing / cluster after next's Leading); simpler: convert the (cluster_id,affinity) pair into an absolute boundary index over the visual cluster order, add/subtract 1, convert back. Handle the leading edge of the first cluster as a distinct boundary.

## 3. [HIGH] `layout/src/text3/cache.rs:6081`
**min-content width computes to 0 for CJK / break-all / overflow-wrap:anywhere text (break-opportunity item advance dropped)**

- **What:** measure_intrinsic_widths accumulates cur_word between break opportunities. When an item IS a break opportunity it resets cur_word to 0 and never adds that item's advance to any word (the else branch that adds `adv` is skipped). For spaces this is fine, but is_break_opportunity_with_word_break returns true for every CJK ideograph in Normal mode (line 10331), for every cluster in break-all, and effectively per-cluster for overflow-wrap:anywhere sizing. So for pure CJK text every character is a break opportunity, cur_word never accumulates any advance, and min_content_width (= max_word) stays 0.
- **Repro:** Give a node inline content "日本語のテキスト" (word-break:normal) and request its intrinsic sizes (e.g. a block/flex-item/table-cell/float with width:min-content, or any min-content contribution path). sizing.rs:707 calls measure_intrinsic_widths; every cluster is a break opportunity so cur_word never accumulates → returns min_content_width = 0.0 → IntrinsicSizes.min_content_width = 0 → box collapses to zero inline width, forcing one ideograph per line / clipping. Identical result for Latin text "hello" with word-break:break-all.
- **Fix:** When an item is a break opportunity that is itself a rendered unit (not a separator/space), include its advance as a one-item word: e.g. `max_word = max_word.max(cur_word).max(if !is_word_separator(item) { adv } else { 0.0 }); cur_word = 0.0;`.

## 5. [HIGH] `layout/src/text3/cache.rs:8563`
**overflow-wrap:normal shreds a long unbreakable word into one grapheme per line instead of overflowing on one line**

- **What:** When the line is empty and the next unit (a whole word with no internal break opportunities) is wider than the line, the OverflowWrap::Normal arm pushes only next_unit[0] (the first cluster) and consumes 1, then breaks the line. On the next line the cursor still holds the remainder of the same word, which is again too wide, so again only its first cluster is emitted. The word is emitted one grapheme-cluster per line. Correct behavior for overflow-wrap:normal is to keep the unbreakable word intact and let it overflow the line box (place the whole unit on one line).
- **Repro:** Shape a single long token with no internal break opportunities, e.g. "Supercalifragilisticexpialidocious" (or a long URL "https://example.com/averylongpathwithnobreaks"), into a container ~40px wide with default styles (overflow-wrap:normal, word-break:normal, hyphens:manual so no hyphenation fires). Call the line-breaking loop that drives break_one_line. Expected (browser/CoreText): one line containing the whole word, overflowing the 40px box. Actual: ~34 lines, each holding a single character stacked vertically (until the trailing suffix finally fits within 40px). Assert on returned line count / that the first line contains the full token.
- **Fix:** In the Normal arm, push the entire next_unit and consume next_unit.len() (place the whole unbreakable unit on this line and let it overflow), matching the intent of the Anywhere/BreakWord branch's progress guarantee but without splitting.

## 6. [HIGH] `layout/src/text3/cache.rs:9830`
**letter-spacing and word-spacing are excluded from line-break width and alignment measurement**

- **What:** `get_item_measure` returns only `cluster.advance + kerning`. Line breaking (break_one_line, cache.rs:8489-8492) sums `get_item_measure` to decide what fits, and `calculate_alignment_offset` (cache.rs:9334-9337) uses the same sum for center/right positioning. Yet letter-spacing and word-spacing are added ONLY later in position_one_line (cache.rs:9301-9315). So the width used for wrapping and alignment omits all inter-character/inter-word spacing.
- **Repro:** Container width = W. Single line of text with N clusters, no wrapping opportunities except at the end, styled `letter-spacing: 4px` (Spacing::Px(4)), and shaped advance sum (advance+kerning) that just fits: sum == W. break_one_line computes unit_width = sum = W <= available = W, so it keeps the line unbroken. position_one_line then advances main_axis_pen by an extra 4px per cluster => rendered width = W + 4*N, overflowing the box by 4*N px (clipping/overlapping the next element). For `text-align:right`: calculate_alignment_offset uses total_width = W (< available), offset = available - W; actual rendered width W+4*N pushes the right edge to available + 4*N, past the right margin. Analogous with `word-spacing` on lines containing space separators.
- **Fix:** Fold per-cluster letter-spacing (and word-spacing on separators) into get_item_measure, or add a spacing term when accumulating unit_width in break_one_line and when summing total_width in calculate_alignment_offset, using the same cursive/word-separator gating as position_one_line.

## 9. [MEDIUM] `layout/src/solver3/getters.rs:2797`
**Sub-pixel letter-spacing / word-spacing silently quantized to whole integer px**

- **What:** text3 stores Spacing as `Px(i32)` (cache.rs:3308, whole integer pixels). Both the fast and slow paths convert CSS spacing via `Spacing::Px(px_val.round() as i32)` (getters.rs:2797, 2809 for letter-spacing; :2822, :2834 for word-spacing). - **Repro:** Lay out a 40-char single-style text run with letter-spacing:0.4px; the resolved Spacing is Px(round(0.4))=Px(0), so per-cluster main_axis_pen += 0 (cache.rs:9305) and total added tracking is 0px instead of ~16px. The correct sub-px value (0.4 / 1.5) is available from compact_cache.get_letter_spacing but discarded by the `.round() as i32` at getters.rs:2797 (fast) / 2809 (slow).
- **Fix:** Change Spacing::Px to hold f32 (keep a hashable wrapper like the existing Em variant uses to_bits), and drop the `.round()` casts; propagate f32 through the position math (cache.rs:7148,9302,etc. already do `px as f32`).
