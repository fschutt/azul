# Fix brief — cluster HINT  (11 confirmed bugs)

Each bug was adversarially verified. Fix the ROOT CAUSE per the sketch; keep the
public API stable unless the fix requires it (note any signature change). After all
fixes, the crate MUST compile.

## 1. [HIGH] `third_party/allsorts/src/hinting/interpreter.rs:484`
**Per-glyph graphics-state reset omits round_state (round mode leaks from prep into every glyph)**

- **What:** At the start of each glyph program the code does `self.gs = self.default_gs.clone()` (the post-prep state) and then re-resets the projection/freedom/dual vectors, rp0-2, loop, and zp0-2. It does NOT reset `round_state`. FreeType's TT_Run_Context resets round_state to 1 (RoundState::Grid) at the start of every glyph, exactly like it resets loop/rp/gep/vectors. Here, whatever rounding mode the prep (CVT) program last set persists into all glyph programs.
- **Repro:** Load a font whose prep (cvt/fpgm-driven) program ends with a non-grid rounding instruction still active — e.g. append RTDG (0x3D), RUTG (0x7C), or SROUND with a non-grid period before prep returns (common in MS VTT-compiled prep that tunes CVT cut-ins). Call execute_prep, then hint_glyph on any glyph that uses MDRP/MIRP/MDAP with the round bit set. Observe self.zones[1].current Y positions snap to the leaked grid phase (e.g. half-grid or double-grid) instead of integer grid, diverging from FreeType v40 output. Contrast with the same run after adding self.gs.round_state = RoundState::Grid at line 485, which restores integer-grid snapping matching FreeType.
- **Fix:** Add `self.gs.round_state = RoundState::Grid;` (and the matching super-round params if you want full parity) to the per-glyph reset block after line 484, matching FreeType's TT_Run_Context.

## 2. [HIGH] `third_party/allsorts/src/hinting/interpreter.rs:990`
**WCVTP/WCVTF resize CVT to an unbounded index → OOM crash on corrupt index**

- **What:** WCVTP (0x44, line 990) and WCVTF (0x70, line 1206) grow the cvt vector with `if i >= self.cvt.len() { self.cvt.resize(i + 1, 0) }` without the 10_000 cap that guards WS/RS (lines 965/978), move_point (1609), and get_point (1656). `idx` is `self.pop()? as u32 as usize`, so a negative or huge stack value becomes a multi-billion index and the resize attempts a multi-gigabyte allocation, aborting the process.
- **Repro:** Malicious font with a TrueType program (fpgm/prep or a glyph's instructions) that pushes a value and a negative/huge index then calls WCVTP: bytes `PUSHW[] 0x00 0x00  PUSHW[] 0xFF 0xFF  WCVTP(0x44)` — the -1 index yields i=4294967295, self.cvt.resize(4294967296, 0) attempts a ~16 GB allocation and aborts the process. Same via WCVTF (0x70). By contrast the same index through RS/WS/RCVT returns HintError::InvalidStorageIndex/InvalidCvtIndex.
- **Fix:** Apply the same bound as WS/RS: `if i > 10_000 { return Err(HintError::InvalidCvtIndex(idx)); }` before resizing, in both WCVTP and WCVTF.

## 3. [HIGH] `third_party/allsorts/src/hinting/interpreter.rs:2561`
**DELTAP pops (arg, point) in the wrong order — point index and delta byte are swapped**

- **What:** In op_deltap the loop pops `arg` first (top of stack) then `point_idx`. But per the MS/Apple spec and FreeType (Ins_DELTAP: `A = stack[args+1]` is the point index = top element, `B = stack[args]` is the exception byte below it), the POINT INDEX is on top and the arg byte below. op_deltac (line 2602) correctly pops the index first then arg — so DELTAP is inconsistent with DELTAC and with FreeType. The stale comment at line 2533 ('arg is on top') encodes the mistaken belief; the DELTAC 'TEST: try reversed pop order' comment shows the index-on-top order was fixed for DELTAC but not DELTAP. Net effect: every DELTAP treats the delta byte as the point number and the point number as the delta byte, so DELTAP-based grid tuning is applied to the wrong point, by the wrong magnitude, at the wrong ppem — DELTAP is effectively broken.
- **Repro:** Font bytecode intending 'move point 20 by +1 delta step at ppem = delta_base+3': push point=20 (0x14), push arg=0x38 (ppem_offset 3, magnitude 8 => +1 step), push n=1, then DELTAP1. Vendored op_deltap pops arg=20 (0x14 => ppem_offset (20>>4)=1 => target ppem delta_base+1, magnitude 20&0xF=4 => -4 steps) and point_idx=0x38=56. So instead of nudging point 20 at ppem base+3, it moves point 56 at ppem base+1 by a negative delta. Running the same pair through op_deltac's (correct) order yields ppem base+3, magnitude +1, index 20 — demonstrating the inconsistency. Fix: swap the two pops in op_deltap to `let point_idx = self.pop()? as u32; let arg = self.pop()? as u32;` and correct the comment at line 2533.
- **Fix:** Swap the two pops so the point index is read first: `let point_idx = self.pop()? as u32; let arg = self.pop()? as u32;` (mirroring op_deltac).

## 4. [MEDIUM] `third_party/allsorts/src/hinting/graphics_state.rs:200`
**SROUND/S45ROUND phase hardcoded to 16/32/48 instead of period/4, period/2, 3*period/4**

- **What:** set_super_round hardcodes phase = 16/32/48 F26Dot6 for phase bits 1/2/3. Per the spec/FreeType, phase must be derived from the actual period: period/4, period/2, 3*period/4. Since period varies (32, 64, or 128 for SROUND; and is multiplied by ~46/64 for S45ROUND at line 193), the hardcoded phase is only correct for period == 64 (1px) in non-45 mode. Threshold at line 210 correctly uses self.super_round_period, but phase does not.
- **Repro:** Execute SROUND with argument byte 0x10: period_bits=(0x10>>6)&3=0 -> period=32 (1/2px); phase_bits=(0x10>>4)&3=1. Spec/FreeType phase = period/4 = 8 F26Dot6, but code sets 16. Any point rounded via super_round() while this state is active is offset by an extra 8/64 (1/8) pixel, snapping it to the wrong sub-grid. Same class of divergence for period=128 (bits 10, phases should be 32/64/96 vs 16/32/48) and any S45ROUND (period=46, phases should be 11/23/34 vs 16/32/48).
- **Fix:** Compute phase from period after the is_45 adjustment: `self.super_round_phase = match phase_bits { 0 => ZERO, 1 => period/4, 2 => period/2, 3 => period*3/4 }` using self.super_round_period.

## 5. [MEDIUM] `third_party/allsorts/src/hinting/interpreter.rs:1290`
**FLIPRGON/FLIPRGOFF iterate lo..=hi with unchecked bounds → hang on large/negative index**

- **What:** FLIPRGON (0x81, lines 1290-1296) and FLIPRGOFF (0x82, 1300-1306) do `let hi = self.pop()? as usize; let lo = self.pop()? as usize; for i in lo..=hi { ... get_mut(i) }`. The per-iteration work is only a bounds-checked get_mut (so no OOB), but the loop count is unbounded and the global instruction_count guard is NOT incremented inside the loop, so a huge range never trips MAX_INSTRUCTIONS.
- **Repro:** Craft a glyph program whose bytecode pushes lowpoint then highpoint and invokes FLIPRGON: e.g. PUSHW[0] 0x00 0x00 (push 0 = lo), PUSHW[0] 0xFF 0xFF (push -1 = hi), then opcode 0x81. In dispatch, hi = (-1i32) as usize = 18446744073709551615, lo = 0, so `for i in 0..=hi` spins ~1.8e19 iterations of get_mut(None) with no instruction_count check -> hang. Same with 0x82 FLIPRGOFF. Fix: reject/skip when hi < lo or hi >= self.zones[1].flags.len(), or clamp hi to flags.len().saturating_sub(1).
- **Fix:** Clamp/validate the range against the glyph zone length and/or reject when hi >= zone.len() or hi < lo, e.g. `let hi = hi.min(self.zones[1].flags.len().saturating_sub(1));` and skip when lo > hi.

## 6. [MEDIUM] `third_party/allsorts/src/hinting/interpreter.rs:1435`
**Stack has no headroom over maxStackElements; under-declared fonts lose ALL hinting**

- **What:** The interpreter stack is capped hard at max_stack = maxp.maxStackElements (push() errors with StackOverflow at len >= max_stack). Storage, CVT, and the twilight zone all grow dynamically past their maxp limits (storage resize at 968, cvt resize at 991, zone get_point/get_original_point/move_point grow to 10_000), but the stack does not. Fonts routinely under-report maxStackElements; FreeType adds headroom.
- **Repro:** Craft/obtain a TrueType font whose maxp.maxStackElements = N but whose fpgm, prep, or a glyph program legitimately reaches N+1 stack elements at some point (e.g. maxStackElements deliberately set one below the true high-water mark, then a PUSHB pushing when N items are already on the stack). Call Interpreter::new(N, ...) then hint. push() returns HintError::StackOverflow at the N+1th element. If the overflow is in fpgm, fpgm_executed=false and every subsequent hint_glyph_full returns the unhinted outline (font-wide loss of hinting); if in a glyph program, that glyph's hint_glyph_with_orus returns Err via `?` at hinting.rs:284 and the caller falls back to the unhinted outline. FreeType renders the same font hinted because it sizes the stack to maxStackElements + 32.
- **Fix:** Allocate/allow max_stack = maxStackElements + slack (FreeType-style, e.g. +32) or grow the stack dynamically like storage/CVT with a sane cap.

## 7. [MEDIUM] `third_party/allsorts/src/hinting/interpreter.rs:1624`
**move_point drops the move AND the touch when freedom·projection == 0; FreeType clamps F_dot_P to 1.0 and still moves+touches**

- **What:** move_point computes dot = (fv·pv) and returns early with no movement and no touched-flag update when dot==0. FreeType guarantees F_dot_P is never zero: when the dot product computes to 0 it substitutes 0x4000 (1.0), so the point is still displaced by distance*freedomVector and still gets its touch tag set.
- **Repro:** Glyph program: set freedom vector along X and projection vector perpendicular along Y so fv·pv == 0 exactly after 2.14 rounding (e.g. SPVTL-perpendicular along a stem then SFVTL along the same line, or an SVTCA X / project-Y arrangement). Then MDRP/MIRP/IP a stem point. In this interpreter move_point computes dot==0 and returns at line 1625: the point is neither displaced nor tagged TOUCHED_X. A later IUP sees it untouched and interpolates it away from its intended position. FreeType clamps F_dot_P to 0x4000, moves the point by distance*fx/0x4000 in X, and sets TOUCHED_X, leaving it pinned.
- **Fix:** Replace `if dot == 0 { return; }` with `let dot = if dot == 0 { 0x4000 } else { dot };` matching FreeType's F_dot_P clamp, so the displacement and touch flags are still applied.

## 8. [MEDIUM] `third_party/allsorts/src/hinting/interpreter.rs:1967`
**MIRP control-value cut-in gated on rounding and ignores zp0==zp1 condition**

- **What:** The CVT cut-in in op_mirp is `if do_round && (dist - orig_dist).abs() > cvt_ci { dist = orig_dist; }`. FreeType applies the cut-in independently of the rounding bit, and ONLY when the reference and target are in the same zone (`exc->GS.gep0 == exc->GS.gep1`). Vendored (a) skips the cut-in entirely for non-rounding MIRP opcodes, and (b) applies it even across zones where FreeType always keeps the CVT distance.
- **Repro:** Font glyph program: SZP0[0] (twilight), place a twilight reference point at rp0 (e.g. via MIAP), then SZP1[1] (glyph zone), then issue a rounded MIRP (bit 2 set, e.g. 0xE4) on a glyph point relative to the twilight rp0, using a CVT stem-width entry whose value deviates from the scaled outline distance (orig_dist) by more than control_value_cut_in (default 17/16 px). Expected (FreeType/CoreText): keeps cvt_dist → uniform stem width. Vendored: since do_round is set and |cvt−orig| > cutin AND the zone check is absent, sets dist = orig_dist → uneven stem. Second case: same setup but a non-round MIRP (0xE0) with |cvt−orig| > cutin — FreeType snaps to orig_dist (cut-in applies), vendored leaves dist = cvt_dist because the block is gated on do_round.
- **Fix:** Match FreeType: apply the cut-in whenever zp0==zp1 (not gated on do_round), and do NOT apply it when zp0!=zp1: `if zp0 == zp1 && (dist - orig_dist).abs() > cvt_ci { dist = orig_dist; }` placed before the rounding step.

## 9. [MEDIUM] `third_party/allsorts/src/hinting/interpreter.rs:2174`
**SHZ marks every shifted point as TOUCHED (FreeType shifts a whole zone untouched), poisoning later IUP**

- **What:** op_shz shifts the entire target zone via move_point(), and move_point() unconditionally inserts TOUCHED_X/TOUCHED_Y (lines 1640-1645). FreeType's Ins_SHZ calls Move_Zp2_Point(..., touch=FALSE) precisely so that a whole-zone shift does not mark points as touched. SHP/SHC correctly touch (touch=TRUE), but SHZ must not.
- **Repro:** Font glyph program: (1) run MDAP/MIAP etc. to touch a couple of control points on a contour; (2) set freedom+projection vectors to Y axis, set rp2, then push zone e=1 and execute SHZ[0] to bulk-shift the whole glyph zone by rp2's displacement; (3) execute IUP[y]. Expected (FreeType/CoreText): the SHZ-shifted points remain untouched, so IUP re-interpolates them relative to the genuinely touched control points. Actual (this code): every point in zone 1 now has TOUCHED_Y set, so op_iup's per-contour touched list contains all points, iup_interp has no untouched gaps to fill, and the points stay at the raw SHZ offset -> different outline. Observable as a per-glyph vertical outline mismatch versus CoreText for any hinted font whose bytecode uses SHZ-before-IUP.
- **Fix:** Give op_shz a direct add-to-current path that does NOT set touched flags (mirror op_shpix's inline move but without the flag inserts), or add a 'touch: bool' parameter to move_point and pass false from SHZ.

## 10. [MEDIUM] `third_party/allsorts/src/hinting/interpreter.rs:2495`
**ISECT treats only exactly-zero determinant as parallel; near-parallel lines extrapolate a far-off point**

- **What:** op_isect only falls back to the midpoint when denom.abs() < 1, i.e. the determinant is exactly zero. For near-parallel line pairs the determinant is a tiny non-zero value (denom is in F26Dot6^2 units, ~2^12 scale, so values like 3..50 are 'almost parallel'). FreeType's Ins_ISECT uses a RELATIVE guard (19*|discriminant| <= |dotproduct|) so shallow-angle intersections are snapped to the midpoint. Here they instead take the else branch and compute t = muldiv(numer,64,denom) with a nearly-zero denom, extrapolating the intersection far outside the two segments.
- **Repro:** Craft/select a glyph whose TrueType bytecode issues ISECT[] on two nearly-collinear-but-offset segments meeting at a shallow angle (~2-3 deg), e.g. a diagonal stem edge (line A) and a near-parallel serif line (line B) that are spatially offset (dp = pb0-pa0 large). Example current-zone coords (F26Dot6): pa0=(0,0), pa1=(1000,50), pb0=(0,400), pb1=(1000,440). Then dax=1000,day=50,dbx=1000,dby=40; denom = 1000*40 - 50*1000 = -10000 (small, non-zero). numer = dpx*dby - dpy*dbx = 0*40 - 400*1000 = -400000; t = ft_muldiv(-400000,64,-10000) = 2560 (=40.0 in 64-scale). x = 0 + ft_muldiv(1000,2560,64)=40000, y=0+ft_muldiv(50,2560,64)=2000 — the intersection is placed at (~625px, ~31px), far outside both 0..1000/~0..440 segments. FreeType: 19*|discriminant| = 19*(10000/64) ≈ 2969 vs |dotproduct| = (1000*1000+50*40)/64 ≈ 15656; since 2969 < 15656, FreeType takes the midpoint = ((0+1000+0+1000)/4, (0+50+400+440)/4) = (500, ~222). Divergence corrupts the contour.
- **Fix:** Detect near-parallel with a relative test like FreeType (compare |dax*dby-day*dbx| against |dax*dbx+day*dby| scaled by a factor), not denom==0.

## 11. [LOW] `third_party/allsorts/src/hinting.rs:256`
**Phantom point pp1.x hardcoded to 0 instead of (xMin - lsb); pp2.x wrong for lsb != xMin**

- **What:** hint_glyph_full appends phantom[0] = (0,0) and phantom[1] = (advance,0). Per the TrueType spec (and calculate_phantom_points in glyf.rs line 572) pp1.x = xMin - lsb and pp2.x = pp1.x + advance, scaled to F26Dot6. The code assumes lsb == xMin (pp1.x == 0). The same hardcoding is repeated in hinted_advance_f26dot6 (lines 333-336) and hint_glyph_with_flags (lines 393-396). Additionally the vertical phantoms pp3/pp4 default to y=0 and the only caller (glyph_cache.rs build_hinted_path) passes phantom_top_y/bottom_y = None, so pp3/pp4 are always (0,0) in production.
- **Repro:** 1) Take/craft a TrueType font with a glyph where hmtx lsb != glyf xMin (e.g. lsb=50 but xMin=0), and whose glyph program does SRP0[pp1] then MDRP the left outline point (a standard LSB-control idiom). 2) Render at a ppem where hinting is active. FreeType: pp1.x = (xMin-lsb)=-50 funits scaled; the MDRP measures lsb (50 funits) from pp1 and rounds it to the grid. 3) This code: pp1.x forced to 0, so the same MDRP measures xMin (0 funits) from pp1 -> grid-fits the wrong distance, shifting the left stem/sidebearing by scaled(xMin-lsb) relative to FreeType. Assert the hinted x of the leftmost outline point differs from FreeType's by ~(xMin-lsb)*scale. With a normalized font (lsb==xMin) the two agree, confirming the offset is exactly (xMin-lsb).
- **Fix:** Thread the scaled (xMin - lsb) into phantom[0].x and set phantom[1].x = phantom[0].x + advance; have the caller compute and pass phantom_top_y/bottom_y from vertical metrics as the doc comment already describes.
