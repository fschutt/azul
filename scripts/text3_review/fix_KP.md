# Fix brief — cluster KP  (4 confirmed bugs)

Each bug was adversarially verified. Fix the ROOT CAUSE per the sketch; keep the
public API stable unless the fix requires it (note any signature change). After all
fixes, the crate MUST compile.

## 1. [CRITICAL] `layout/src/text3/knuth_plass.rs:420`
**Knuth-Plass never breaks a paragraph that ends in a Box (no trailing forced break) — whole paragraph collapses to one line**

- **What:** find_optimal_breakpoints only updates breakpoints[i+1] when nodes[i] is a Penalty. The backtrack starts at current = nodes.len() (== n) and reads breakpoints[n]. breakpoints[n] is only ever assigned if the LAST node (index n-1) is a Penalty. But convert_items_to_nodes emits a trailing Penalty only after spaces, hyphens, objects, ZWSP or explicit Break — a paragraph that ends in an ordinary word emits Box nodes as its final nodes. So breakpoints[n] stays at its init value {demerit: INFINITY, previous: 0, line: 0}. The backtrack then produces breaks == [n], i.e. one single line spanning the entire paragraph. Standard Knuth-Plass appends a forced end-of-paragraph penalty (glue + \penalty -inf) to anchor the final line; this code never does.
- **Repro:** Call perform_fragment_layout with fragment_constraints.text_wrap = TextWrap::Balance, available_width = Definite(W) narrower than the text, and text "Hello there world" (no trailing space/newline). shaped_items end in Cluster('d') → nodes end in LayoutNode::Box. breakpoints[nodes.len()] is never assigned, backtrack returns breaks=[n], and every glyph is positioned on line 0 — one overflowing line instead of balanced wrapped lines. Contrast: "Hello there world " (trailing space) wraps correctly because the final node is a zero-penalty Penalty, so breakpoints[n] gets set. Fix: after building nodes, append LayoutNode::Glue{width:0, stretch:large, shrink:0} then LayoutNode::Penalty{width:0, penalty:-INFINITY_BADNESS}.
- **Fix:** After building nodes, append a terminal sequence: Glue{width:0, stretch: large, shrink:0} then Penalty{width:0, penalty:-INFINITY_BADNESS} so the final break is forced at n; or special-case the backtrack to seed the last line from the min-demerit reachable index when breakpoints[n] is unset.

## 2. [MEDIUM] `layout/src/text3/knuth_plass.rs:163`
**Knuth-Plass never creates a soft-wrap opportunity after a mid-word hyphen (U+002D/U+2010)**

- **What:** The dedicated hyphen arm (lines 145-159) that emits a zero-width penalty after a cluster ending in '-'/U+2010 only fires when the hyphen cluster is the FIRST item of a run. In practice hyphens occur mid-word, and the general ShapedItem::Cluster arm (line 160) greedily consumes ALL following clusters — including the hyphen cluster — into one word via `while item_iter.peek() is Cluster`. So in a word like "well-being" the '-' is absorbed into the word and no break penalty is inserted after it, contrary to UAX#14 (a hyphen is a break opportunity after it).
- **Repro:** Feed convert_items_to_nodes a cluster sequence for "mother-in-law" (13 per-grapheme ShapedItem::Cluster items, the two '-' clusters have text=="-") with hyphenator=None. Observe the produced Vec<LayoutNode>: it is 13 consecutive Box nodes with zero Penalty/Glue nodes between them — no zero-width Penalty follows either '-'. Then in a container narrower than the token's advance, kp_layout has no feasible breakpoint inside the token and the line overflows instead of wrapping at "mother-"/"in-". Contrast: place a single leading hyphen right after a space ("x -y") and arm 145 DOES fire, proving the arm works only at run head.
- **Fix:** In the general cluster-collection loop, stop collecting (or emit a zero-width Penalty) after any cluster whose text ends with U+002D or U+2010, so intra-word hyphen break opportunities are preserved.

## 3. [MEDIUM] `layout/src/text3/knuth_plass.rs:460`
**Knuth-Plass includes the line-terminating space in line width and justification gaps**

- **What:** In position_lines_from_breaks a line is nodes[start..end] where end is the break index just past the Penalty that follows the trailing Glue (space). The trailing Glue's item is thus included in line_items. line_width (line 460) therefore includes the trailing space, should_justify counts it as a justification gap (space_count at line 491-494 includes it), and extra_per_space is distributed into the trailing space too. There is no trailing-space trimming as in the greedy path (break_one_line line 8578).
- **Repro:** Lay out a block with definite width, text-wrap: balance and text-align: justify containing a paragraph that wraps to 2+ lines (e.g. "the quick brown fox jumps over the lazy dog ..."). This enters kp_layout (cache.rs:7991). For each non-final line, the trailing space is included in line_width and space_count, so extra_per_space is distributed into it and the last word's right edge lands at available_width minus a stretched space width — the word fails to reach the right margin and inter-word spacing is slightly tighter than a correct justify. Compare against the same text with text-wrap: normal (greedy path), which trims the trailing space and reaches the margin.
- **Fix:** Before measuring/justifying a line in position_lines_from_breaks, drop trailing word-separator items from line_items (or exclude the final glue from line_width and the space count).

## 4. [LOW] `layout/src/text3/knuth_plass.rs:290`
**Knuth-Plass MinContent produces a single line, so min-content == max-content for text-wrap:balance**

- **What:** find_optimal_breakpoints maps AvailableSpace::MinContent to line_width = f32::MAX/2 (identical to MaxContent). The comment claims 'let the greedy line breaker break after each word', but this is the KP breaker with no greedy fallback — with an effectively infinite width the DP puts the whole paragraph on one line. position_lines_from_breaks then sets available_width_f32 = line_width (the full paragraph width). So a text-wrap:balance element reports min-content width equal to its max-content width.
- **Repro:** Construct an inline paragraph with several words whose total single-line advance greatly exceeds the widest single word (e.g. "internationalization considered harmful today") and set text-wrap: balance. Invoke kp_layout with constraints.available_width = AvailableSpace::MinContent (as the table auto-layout path does via measure_cell_min_content_width -> calculate_layout_for_subtree in ComputeSize with MinContent). find_optimal_breakpoints returns a single break spanning all nodes; position_lines_from_breaks yields one line whose width = sum of all advances (== the MaxContent result). Expected CSS min-content = width of the widest unbreakable word ("internationalization"). Observed = full paragraph width, so the table column/cell is sized to max-content and never shrinks to the widest-word width.
- **Fix:** For MinContent in the KP path, break at every legal opportunity (line_width -> 0 or a per-word greedy pass) and report the widest resulting line, rather than reusing the MaxContent infinite width.
