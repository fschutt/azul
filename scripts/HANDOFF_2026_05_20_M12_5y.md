# M12.5y Handoff — 2026-05-20 — node_count was a PHANTOM; real bug = apply_ua_css returns via its error path without restoring SP

## TL;DR (supersedes the M12.5x handoff entirely)

The whole prior investigation (push_to slot store, u128, NEON-Q, grow_one, frame-overlap,
"self(X0)-relative store", "node_count@self+0") chased a **phantom**. A Heisenbug-proof
**lift tracer** (built this session) proved:

- **node_count is NEVER corrupted.** `CssPropertyCache` is `#[derive]`-only (no `repr(C)`),
  so node_count is at **offset 0x188**, not 0. empty() writes node_count=1 and nothing
  overwrites it. The cache at `0x2ee10` is built fine.
- The real bug: in `StyledDom::create_from_compact_dom`, the pointer to its local
  `css_property_cache` is correct (`0x2ee10`) after empty()+restyle (`[8]=1`) but **NULL/garbage
  after the `apply_ua_css` call** (`[9]=0`). create_from addresses that local **SP-relative**
  (`X0 = SP+8416` at the call site) and also caches it in a callee-saved reg, so a corrupted
  SP/callee-saved after the call makes every later cache read — including `build_compact_cache`'s
  `self` — resolve to a wrong/NULL address. That wrong `self` is the real **768 MiB OOB**.
- **Why SP is corrupted:** the SP trajectory shows `apply_ua_css` (lifted `sub_a47e3c`) executes
  ONLY its prologue `SP -= 0x1b0`; its **epilogue SP-restore never runs**. opt kept the SP restore
  in block `%29` (the normal epilogue), but the runtime returns via block `%64` (and/or `%600`),
  which tail-calls `raw_vec::handle_error` then `-> common.ret` **without restoring SP/callee-saved**.
- **Root cause class:** `apply_ua_css` is taking an **allocation-error / panic exit** (block %64 =
  `bl raw_vec::handle_error`; reached via the alloc-result check `%cmp.i.i1275 = icmp eq <alloc
  result>, 0`, or the `%600` branch). Natively these handlers are **noreturn** (they abort), so that
  path never restores SP — which is fine natively because it aborts. But the lift **stubs the
  noreturn handlers (`handle_alloc_error` = Leaf, `handle_error`) to RETURN**, so lifted apply
  RETURNS via the error path with **unrestored SP**, silently corrupting the caller. Either (a) a
  mis-lifted condition makes apply take the error path when native wouldn't (the prime suspect: the
  BumpAlloc result is mis-tracked as 0 → alloc-success check fails), or (b) the noreturn stubs
  should `unreachable`/trap instead of returning (would at least convert silent corruption into a
  loud trap and expose (a)).

## The Heisenbug-proof tracer (the key tool — KEEP)

`dll/src/web/transpiler_remill.rs::inject_store_logging` (+ `parse_logged_write`,
`parse_memintrinsic_dest/_len`), gated by env `AZ_LOG_STORES` (`=ALL` instruments every
non-wrapper dep; or comma-list of fn-name substrings). In the **subprocess** pipeline (active:
`use_native_remill()` needs `AZ_NATIVE_REMILL` which is unset, so `.opt.ll` files exist), AFTER
`opt -O2` and BEFORE `llc`, it injects before each `store`/`memset`/`memcpy`:
`%azp=ptrtoint <dest> to i32; <val materialization>; call @__az_logst(addr,id,val)` plus an appended
`@__az_logst(addr,id,val)` that writes `(addr,id,deptag,val)` 16-byte entries to a ring buffer at
wasm linear **0x41000** when `addr ∈ [0x2cb00,0x2f000)` (the apply/cache page) OR `addr==0xF0000`
(the **SP-trajectory marker**: a `store ..., ptr %SP` is logged with addr 0xF0000 and val=the SP
value). **Heisenbug-proof: store addresses are fixed SSA after opt; only llc reg-alloc runs after,
which can't change a computed address — so it observes the real corrupting writes without
perturbing the AArch64 lift.** Saved as `<stem>.instr.ll`. Read via `scripts/m9_e2e/baseline-probe.js`
(reads 0x41000; prints node_count bracket, the multi-point `&node_count` address consistency, the
SP trajectory, cache-struct writes). node_count addr captured by `AZ_DBG_NC[41/43/44/45]` in
styled_dom.rs ~1053.

- `&node_count [7t]=0x2ef98  [9t]=0x0  [11t]=0x0  &cache[dump]=0x0` → cache base correct after
  empty, NULL after apply.
- SP trajectory: `apply_ua_css (dep sub_a47e3c)` appears once: `id=0 SP=0x2cb80` (prologue only).
  Balanced fns (e.g. grow_one) show prologue+epilogue. apply's block %29 SP-restore (id=28) never

  (`old @__az_bump_ptr`); `br %cmp.i.i1275, label %64(handle_error), label %69`. Also
  `br %cmp.i53.i.i2835, label %600, label %603` and `%600 -> %64` unconditionally.
