## TL;DR

The M12 cascade blocker is **still open**. The cascade
(`StyledDom::create`) runs in wasm but its output struct at
`styled_ptr` reads back **all-zero** (only a manually-injected
`0xCAFEBABE` marker at +8 survives). The M11 styled-dom-hydrate
gate **passes** but with `node_data.len()=0` annotated as a known
issue.

This session landed **real lift-pipeline correctness fixes** and
**narrowed the root cause** to a specific trigger, but did NOT
unblock the cascade. One promising fix (`strip_noalias`) was tried
and **disproven** ‚Äî see below.

## What actually got fixed (real, committed, verified)

1. **`1fc93c58d` ‚Äî LDR Q mirror false-match (the big one).**
   `scan_arm64_adrp_accesses` had wrongly extended LDUR-Q detection
   to `top8 == 0x3D`, but `0x3D` is LDR-scaled, not LDUR. For LDR Q
   whose `imm12 % 4 == 0` (every other 16-byte const-pool stride),
   the LDUR arm fired and recorded a bogus mirror range at the
   sign-extended imm9 instead of `imm12*16`.
   **Effect:** `make_test_struct` heap went from 32/64 ‚Üí **64/64**
   correct values. This proves single-level sret works.

2. **`6784b1bf2` ‚Äî LDP/STP Q/D/S + LDUR Q mirror coverage.**
   Extended `scan_arm64_adrp_accesses` with the full SIMD&FP
   pair-load/store family (signed-offset, pre/post-index).

3. **`4dbbeed81` ‚Äî helper-IR `store volatile`.**
   `__remill_write_memory_*` helpers now emit `store volatile` so
   LLVM DSE can't drop lifted writes that have no in-function
   reader (they're observed externally by JS / sibling fns).

4. **`329b4cc05` ‚Äî wasm32 IR retarget.**
   `retarget_to_wasm32()` rewrites the AArch64 `target triple` /
   `datalayout` to wasm32 before opt/llc, so SROA/InstCombine run
   with the correct (matching) target.

5. **`9df35fab9` ‚Äî `AZ_OPT_LEVEL` env knob** (O0/O1/O2/Oz) for
   pipeline debugging.

6. **`4e302892a` ‚Äî `strip_noalias_from_sub_args` pass.** Removes
   `noalias` from `@sub_*` defines/declares. **NOTE: this did NOT
   fix the corruption** (see "Disproven" below) ‚Äî it's a harmless
   defensive change but the next agent should consider reverting or
   keeping it as-is depending on whether they pursue the
   attribute-stripping angle.

All 7 functional M11 gates remained **GREEN** throughout. Bundle
size net change ‚âà +665 bytes (~0.14%).

## The root-cause narrowing (verified by bisect)

Built reproducer functions in `eventloop.rs` (`make_test_struct*`)
and probed their Box::new'd heaps via JS. Key results:

| Pattern | Lifted? | make_test_struct heap |
|---------|---------|----------------------|
| `() -> Struct` (no args) | yes | ‚úì 64/64 |
| `(u32) -> Struct` (by-value arg) | yes | ‚úì 64/64 |
| `(&mut u32) -> Struct` (no inner write) | yes | ‚úó corrupted |
| `(&mut u32) -> ()` (void return) | yes | ‚úó corrupted |

**Decisive finding (commit `667215072` + `62a4ada71`):** merely
**LIFTING** a function whose signature contains a `&mut T` arg
corrupts the const-pool reads of *other, unrelated* lifted
functions. The corruption is at **LIFT/BUILD time, not runtime** ‚Äî
proven by gating the call behind an always-false branch (`bl`
present in native, never executed at runtime) and still seeing
`make_test_struct` break.

By-value args do NOT trigger it. The `&mut` arg alone is the
trigger (independent of whether the body writes through it, and
independent of return type).

This explains why the cascade is the only failing path: every
function in the `StyledDom::create(dom: &mut Dom, css) -> Self`
transitive chain that takes a `&mut T` arg poisons the build.

## Disproven this session

- **`noalias` is NOT the (sole) trigger.** The `strip_noalias`
  fix (`4e302892a`) removed `noalias` from sub_* args but a
  re-added `&mut`-arg control function STILL corrupted
  `make_test_struct`. So Rust's `&mut` lowers to MORE than just
  `noalias` ‚Äî likely also `dereferenceable(N)`, `align(N)`,
  `nonnull`, and these (or remill's handling of the `ptr` arg
  itself) are still triggering the cross-function corruption.

- **Earlier "ROOT CAUSE: sret breaks with &mut arg" (reverted
  `c17a946de`) was MISDIAGNOSED.** The "6/64 non-zero" heap I saw
  was from a *trapped* hydrate (too many cumulative Box::new test
  allocs ‚Üí bump-alloc exhaustion / trap), not a real sret bug. Be
  careful: piling many reproducer Box::new calls into hydrate
  traps it. Test ONE reproducer at a time.

## What is NOT the cause (ruled out earlier this session)

- Multi-level sret chains (verified working up to 5 levels ‚Äî but
  these were before the cumulative-trap discovery, so re-verify
  with single-reproducer builds if depended on)
- LTO specialization (tested `AZ_WASM_DEBUG=1` ‚Üí `--lto-O0`, same
  zero output)
- opt level (O0/O1/O2/Oz all produce same structure)
- Plain heap alloc, Vec push+drop, multi-Vec ‚Äî but again, verify
  these in isolation, not stacked.

## Exact reproducer recipe for next agent

1. At clean HEAD, `make_test_struct` heap = 64/64 correct.
2. Add ONE function `fn f(x: &mut u32) -> TestStruct256` with
   `#[inline(never)] #[no_mangle]`.
3. Reference it in hydrate behind `if black_box(state) ==
   0xDEADBEEF { Box::new(f(&mut a)); }` (always-false; lifted but
   never run).
4. Rebuild, probe `make_test_struct` heap ‚Üí it will now be
   corrupted (mostly zeros + `0x8`/`0x1` garbage).
5. Remove the function ‚Üí `make_test_struct` heals.

This is a deterministic build-level reproducer. Use it to bisect
the exact LLVM IR attribute or remill lift behavior.

## Recommended next steps (priority order)

1. **Dump and diff the lifted IR.** Build with and without the
   `&mut`-arg control fn (use `AZ_REMILL_KEEP_SCRATCH=1`, but
   `rm -rf` old scratch dirs first ‚Äî see disk note). `diff` the
   `make_test_struct.lifted.ll` between the two builds. It should
   be byte-identical (same fn). If the corruption is at link/opt,
   diff the `azul-mini` `.opt.ll` / wasm `.wat` instead. Find what
   the `&mut`-arg fn changes about make_test_struct's emitted
   const-pool loads.

2. **Test attribute-stripping incrementally.** Extend
   `strip_noalias_from_sub_args` to also strip `dereferenceable`,
   `align`, `nonnull`, `readonly`, `writeonly` from sub_* args.
   Rebuild + re-run the reproducer. If make_test_struct heals,
   you've found the trigger attribute(s).

3. **If attributes aren't it: suspect the mirror.** The `&mut`
   fn's adrp+ldr targets get added to `accessed_ranges`. Maybe
   they overlap/displace make_test_struct's ranges in
   `build_mirror_segments` (the zero-trim or page-merge logic).
   Enable `AZ_DEBUG_MIRROR_ALL=1` (the debug eprintln was reverted
   but easy to re-add) and compare mirrored byte ranges for
   make_test_struct's page between the two builds.

4. **Fallback to unblock Phase 5:** if the cascade can't be fixed
   quickly, construct a minimal `StyledDom` in Rust source via raw
   field writes (no `&mut`-arg helper in the chain) so
   `getStyledDomNodeCount` returns ‚â• 1, then proceed to lift
   `LayoutWindow::layout_dom_recursive` (M12.6 / Phase 5) with that.

## Disk-space caution (caused a crash this session)

The transpiler writes a scratch dir per server run at
`/var/folders/.../T/azul-web-transpiler-<pid>/` (~15-17 MB of
lifted IR each). With `AZ_REMILL_KEEP_SCRATCH=1` they accumulate
and filled the disk. Mitigations:
- `rm -rf /var/folders/*/T/azul-web-transpiler-*` before each
  KEEP_SCRATCH run.
- Don't set KEEP_SCRATCH unless actively inspecting IR (without
  it, the transpiler auto-wipes on drop).
- `target/debug` (7+ GB) is not used by the web build ‚Äî safe to
  `rm -rf` if space is tight.
