## What's mocked / known-broken

### 1. Eventloop runtime behaviour (the M8.4b stall)

The lifted `AzStartup_init` body returns a non-zero value but
doesn't actually populate `EVENTLOOP_PTR`. Subsequent
`AzStartup_registerCallback` traps with "memory access out of
bounds". The lift produces correct-*shape* bytes (8 exports + right
signatures) but execution-broken bodies for any function that
touches `Box::new`, `BTreeMap::insert`, atomic-pointer reads, or
static-variable references.

Root cause: lifted code references *native dylib addresses* for
globals/heap-allocated objects; those addresses are meaningless in
wasm linear memory. `inttoptr i64 %native_addr to ptr` truncates to
a garbage wasm offset.

Same family as `memory/m6_sroa_gap_2026_05_18.md`.

### 2. Layout callback return shape

M8.3 lifts the layout callback under the canonical Callback wrapper
shape — three Wreg args + Wreg return. The real `AzStyledDom` return
(>16B aggregate) uses AArch64's X8 hidden-pointer convention, which
the canonical wrapper doesn't seed. Calling the lifted layout from
JS would write to wherever X8 happens to be (uninitialized = 0 =
trap on store).

M8.5 was supposed to introduce a `Pcs::HiddenPtrReturn` variant +
heap-allocated return buffer + serialize the StyledDom (per Q1 user
direction). M8.5 has the same constraint as M8.4b — Rust runtime
primitives don't lift.

### 3. Per-callback framework calls still noop'd

M7's intercept pass replaces every `bl <framework_fn>` with a noop
stub. on_click's `MyDataModel_downcastMut` call returns the memory
token unchanged. Counter doesn't actually increment.

M8.9 was supposed to fix this by routing framework calls to
`azul-mini.wasm` imports. Same constraint family as M8.4b.

### 4. JS-side listener.js not yet rewritten

M8.6 was supposed to rewrite loader.js to:
- Instantiate mini + layout + per-cb modules into a shared
  WebAssembly.Memory + funcref Table.
- Wire native event listeners on root.
- Marshal events into 256B buffers + call AzStartup_dispatchEvent.
- Drain TLV patches + apply to DOM.

Not started. Per-callback URLs are emitted in the HTML
(`data-az-wasm="/az/cb/..."`) but the M0-M7 loader.js doesn't
actually instantiate anything.

## The architectural decision needed

The 2026-05-19 user decision was:

> Eventloop is lifted from libazul via the same M5-M7 remill
> pipeline. NOT a separately-compiled wasm32 crate.

That was a sound decision *for framework functions* (AzDom_addChild,
AzString_clone, etc. — code that only touches register-flowed
pointers).

It does NOT work for *runtime primitives* (alloc, atomics,
collection internals, static-variable references). The eventloop is
made of runtime primitives.

Three options documented in
`memory/m8_4_lift_runtime_gap_2026_05_16.md`:

**A — Hand-write the eventloop runtime in LLVM IR (RECOMMENDED).**
Inline string literal in `transpiler_remill.rs` defining all 7
AzStartup_*. Uses static linear-memory storage + bump allocator +
linear-scan callback table. Bypasses the lift entirely for these
functions. M5-M7 user-callback lifts are unaffected. ~6h.

**B — Lifted with full Rust runtime in helper IR.** Extend
`emit_helper_ir` with bump-allocator bodies for `__rust_alloc`,
single-threaded `__remill_atomic_*` bodies, AND a custom
LLVM-IR-rewriting pass that translates native-address references
into wasm-ld symbol lookups. 2-3 days.

**C — Stop. Ship M8.1-M8.4a.** Don't make the demo work
end-to-end. The infrastructure is in place but the counter doesn't
increment. Zero additional effort.
