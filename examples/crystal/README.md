# Azul — Crystal binding example

`hello-world.cr` is the counter example. It `require`s the generated
binding as a sibling file:

```crystal
require "./azul"
```

so the generated `azul.cr` must live next to it:

```
examples/crystal/
├── hello-world.cr      # the driver (this file)
└── azul.cr             # generated by azul-doc codegen (lib LibAzul)
```

`azul.cr` is a build artifact — generate it with
`cargo run -p azul-doc codegen all` (it lands in `target/codegen/azul.cr`)
and copy it next to `hello-world.cr`, or download it from a release.

The binding is archetype-A (C-ABI-direct): a single `@[Link("azul")] lib
LibAzul` block declares every struct by value, every enum with an
explicit backing integer, every tagged union as a Crystal `union`, every
callback typedef as a `Proc` alias (`(AzRefAny, AzCallbackInfo) ->
AzUpdate`), and every exported symbol as a `fun` binding. The Crystal-side
`fun` name lowercases the first letter of the C symbol (`AzApp_create`
→ `azApp_create`) so it is a legal method name; the C symbol is kept
verbatim after `=`, so the linker resolves the same exports the C / Zig /
Odin bindings do.

## Build

With `libazul.{so,dylib}` / `azul.dll` in this directory:

```sh
# linux
crystal build hello-world.cr --link-flags "-L."
LD_LIBRARY_PATH=. ./hello-world

# macos
crystal build hello-world.cr --link-flags "-L. -framework Foundation -framework AppKit -framework OpenGL -framework CoreGraphics -framework CoreText"
DYLD_LIBRARY_PATH=. ./hello-world

# windows
crystal build hello-world.cr --link-flags "/LIBPATH:."
hello-world.exe
```

Callbacks are C-direct: `ON_CLICK` / `LAYOUT` are plain non-capturing
`->(...) { ... }` procs passed straight to the C-ABI setters — no
host-invoker, exactly like the C / Zig / Odin bindings. The procs reach
their helpers (`MyData.downcast`, `MyData.type_id`) through *constant*
lookups so they never capture an outer local, which keeps them
non-closure — a hard requirement for passing a Crystal proc as a bare C
function pointer.
