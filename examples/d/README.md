# Azul — D binding example

`hello-world.d` is the counter example. It imports the generated binding
module:

```d
import azul;
```

so the generated `azul.d` (module `azul`) must sit next to it and be
compiled together with the driver:

```
examples/d/
├── hello-world.d        # the driver (this file)
└── azul.d               # module azul (generated by azul-doc codegen)
```

`azul.d` is a build artifact — generate it with
`cargo run -p azul-doc codegen all` (it lands in `target/codegen/azul.d`)
and copy it next to `hello-world.d`, or download it from a release.

## Build

With `libazul.{so,dylib}` / `azul.dll` in this directory:

```sh
# linux (dmd)
dmd hello-world.d azul.d -L-L. -L-lazul -of=hello-world
LD_LIBRARY_PATH=. ./hello-world

# linux (ldc2)
ldc2 hello-world.d azul.d -L-L. -L-lazul -of=hello-world
LD_LIBRARY_PATH=. ./hello-world

# macos (dmd) — link the system frameworks libazul needs
dmd hello-world.d azul.d -L-L. -L-lazul \
    -L-framework -LFoundation -L-framework -LAppKit \
    -L-framework -LOpenGL -L-framework -LCoreGraphics -L-framework -LCoreText \
    -of=hello-world
DYLD_LIBRARY_PATH=. ./hello-world

# windows (dmd)
dmd hello-world.d azul.d -L/LIBPATH:. azul.lib -of=hello-world.exe
hello-world.exe
```

`-L-L.` passes `-L.` to the linker (search the current directory) and
`-L-lazul` passes `-lazul`. The `LD_LIBRARY_PATH=.` / `DYLD_LIBRARY_PATH=.`
prefix is needed at run time because the binary embeds no rpath.

## How it works

Callbacks are C-direct: `on_click` / `layout` are plain `extern(C)`
functions, and their address (`&on_click`, `&layout`) is passed straight
to the C-ABI setters — no host-invoker, exactly like the C, Zig and Odin
bindings. A D function declared `extern(C)` **is** a real C function
pointer.

This binding is experimental and CI-validated (it compiles and the
end-to-end test clicks the counter from 5 to 6 to 8).
