# Azul — V (vlang) binding example

`hello-world.v` is the counter example (`module main`). It imports the
generated binding as a subpackage:

```v
import azul
```

so the generated `azul.v` must live in an `azul/` subdirectory next to it:

```
examples/v/
├── hello-world.v        # module main (this driver)
└── azul/
    └── azul.v           # module azul (generated by azul-doc codegen)
```

`azul.v` is a build artifact — generate it with
`cargo run -p azul-doc codegen all` (it lands in `target/codegen/azul.v`)
and copy it to `azul/azul.v`, or download it from a release into
`azul/azul.v`.

## Build

With `libazul.{so,dylib}` / `azul.dll` in this directory, build the whole
directory (both `.v` files) with `v run .`:

```sh
# linux
LD_LIBRARY_PATH=. v run .

# macos
DYLD_LIBRARY_PATH=. v run .

# windows (azul.dll in the current directory)
v run .
```

`azul.v` carries the link flags (`#flag -L.` + `#flag -lazul`, plus the
macOS `#flag darwin -framework …` set), so V's C backend links `libazul`
automatically. The `LD_LIBRARY_PATH=.` / `DYLD_LIBRARY_PATH=.` prefix is
needed at run time because the binary embeds no rpath.

To produce a standalone binary instead of `v run`:

```sh
v -prod . -o hello-world
LD_LIBRARY_PATH=. ./hello-world     # macOS: DYLD_LIBRARY_PATH=.
```

Callbacks are C-direct: `on_click` / `layout` are plain top-level V
functions (which compile to real C functions) passed straight to the
C-ABI setters — no host-invoker, exactly like the C, Zig and Odin
bindings.

> **Experimental / CI-validated.** The V binding is an archetype-A
> (C-ABI-direct) backend, off the front-page tab set. It is exercised by
> CI rather than hand-maintained; report issues upstream.
