# Azul — Julia binding example

`hello-world.jl` is the counter example. It loads the generated binding as
a submodule:

```julia
include(joinpath(@__DIR__, "azul", "azul.jl"))
using .Azul
```

so the generated `azul.jl` must live in an `azul/` subdirectory next to it:

```
examples/julia/
├── hello-world.jl       # driver (this file)
└── azul/
    └── azul.jl          # module Azul (generated by azul-doc codegen)
```

`azul.jl` is a build artifact — generate it with
`cargo run -p azul-doc codegen all` (it lands in `target/codegen/azul.jl`)
and copy it to `azul/azul.jl`, or download it from a release into
`azul/azul.jl`.

## Run

Julia is interpreted/JIT-compiled — there is no separate build step. Point
`AZUL_LIB` at the native library (`ccall` `dlopen`s it at call time) and run
the script:

```sh
# linux
AZUL_LIB=$PWD/libazul.so julia hello-world.jl

# macos
AZUL_LIB=$PWD/libazul.dylib julia hello-world.jl

# windows (PowerShell)
$env:AZUL_LIB = "$PWD\azul.dll"; julia hello-world.jl
```

If `AZUL_LIB` is unset, the binding falls back to the plain library name
(`libazul` / `azul`), which the dynamic loader resolves via
`LD_LIBRARY_PATH` / `DYLD_LIBRARY_PATH` / `PATH`.

Requires a recent **Julia** (1.6+). No packages are needed — the binding is
self-contained (`ccall` / `@cfunction` / `@enum` are all in `Base`).

## How it works

- Callbacks are C-direct: `@cfunction(on_click, AzUpdate, (AzRefAny,
  AzCallbackInfo))` mints a real C function pointer passed straight to the
  C-ABI setters — no host-invoker, exactly like the Odin and C bindings.
- Every `Az*` type is an **isbits** Julia `struct` (or `@enum`), so it is
  passed by value across `ccall` with the C ABI.
- isbits structs are immutable, so the window is customized with the
  `setfields` functional-update helper rather than field assignment.
- `RefAny` is Azul's refcounted, type-erased box: `AzRefAny_newC` copies
  the model bytes into a heap allocation; `AzRefAny_getDataPtr` recovers a
  `Ptr{MyDataModel}` for in-place mutation on click.
