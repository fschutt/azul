# Azul — Odin binding example

`hello-world.odin` is the counter example (`package main`). It imports the
generated binding as a subpackage:

```odin
import azul "azul"
```

so the generated `azul.odin` must live in an `azul/` subdirectory next to it:

```
examples/odin/
├── hello-world.odin      # package main (this driver)
└── azul/
    └── azul.odin         # package azul (generated by azul-doc codegen)
```

`azul.odin` is a build artifact — generate it with
`cargo run -p azul-doc codegen all` (it lands in `target/codegen/azul.odin`)
and copy it to `azul/azul.odin`, or download it from a release into
`azul/azul.odin`.

## Build

With `libazul.{so,dylib}` / `azul.dll` in this directory:

```sh
# linux
odin build . -out:hello-world -extra-linker-flags:"-L."
LD_LIBRARY_PATH=. ./hello-world

# macos
odin build . -out:hello-world -extra-linker-flags:"-L. -framework Foundation -framework AppKit -framework OpenGL -framework CoreGraphics -framework CoreText"
DYLD_LIBRARY_PATH=. ./hello-world
```

Callbacks are C-direct: `on_click` / `layout` are plain `proc "c"` values
passed straight to the C-ABI setters — no host-invoker, exactly like the
C and Zig bindings.
